# Мини-блок: SQL для AI Engineer (стартовый предэтап)

*Дополнение к дорожной карте. **Стартовый предэтап** — проходится первым. Слой: **[A]** (рабочий уровень). Длительность: **~8–15 часов** (а с курсом Практикума — весь первый месяц с запасом).*

**Зачем (коротко):** это не отдельная профессия, а инженерная гигиена на твоём же критическом пути. pgvector — это Postgres, и поиск по вектору в RAG — это обычный SQL-запрос с оператором расстояния и `WHERE`-фильтром. Агентам часто дают инструмент «сходи в базу» (text-to-SQL). Логи eval и трейсы тоже обычно SQL-запрашиваемые. Вакансии почти всегда подразумевают базовый SQL.

**Когда проходить:** **первым, на старте** — потому что у тебя первый месяц в Яндекс.Практикуме это плотный SQL. Идеальное совпадение: проходишь курс Практикума как основной материал, а этот блок используешь как **карту целей** (что именно нужно для AI Engineer) и как **чек-лист** — отмечаешь подтемы по мере прохождения. Раздел про pgvector (S.7) пройди отдельно: его в обычном SQL-курсе не будет, но именно он связывает SQL с твоим RAG из Этапа 4.

**Как сочетать с Практикумом:**
- S.1–S.6 (выборки, агрегаты, JOIN, CTE, изменение данных) — это ядро любого SQL-курса; Практикум их закроет. Просто отмечай в трекере.
- S.7 (pgvector) и S.8 (text-to-SQL как агентный паттерн) — **специфика AI Engineer**, которой в общем курсе скорее всего не будет. Удели им отдельную пару сессий по ресурсам ниже.

**Уровень, который нужен (этого хватает с запасом):** уверенно читать и писать выборки, агрегаты, джойны, CTE; понимать pgvector и паттерн text-to-SQL. Глубокая оптимизация, оконные функции на уровне виртуоза, хранимые процедуры, администрирование — **не нужны** (территория data-инженеров/DBA).

**Где запускать:** всё — в браузере (тренажёр / Практикум) и локально в Postgres. GPU не нужна. 🍎

## Подтемы

**S.1 Выборка** `[A]` — `SELECT`, `FROM`, `WHERE`, `ORDER BY`, `LIMIT`, `DISTINCT`. Прямая аналогия с `df[...]`/`sort_values`/`head` из Pandas.

**S.2 Фильтрация, операторы и пропуски** `[A]` — сравнения, `AND`/`OR`/`NOT`, `IN`, `BETWEEN`, `LIKE`, `IS NULL`; **обработка пропусков** — `COALESCE`/`NULLIF`, заполнение и отсев `NULL` (базовая чистка данных под датасет/фичи — в Практикуме это отдельный блок «работа с пропусками»).

**S.3 Агрегация** `[A]` — `GROUP BY`, `COUNT`/`SUM`/`AVG`/`MIN`/`MAX`, `HAVING` (фильтр по агрегату). Это `groupby` из Этапа 0 на языке SQL.

**S.4 Соединения (JOIN)** `[A]` — `INNER JOIN`, `LEFT JOIN`, `RIGHT JOIN`, `FULL JOIN`, self-join; связь таблиц по ключам; чем left отличается от inner; особенности соединения (дубли ключей → размножение строк, декартово произведение при отсутствии условия).

**S.5 Подзапросы и CTE** `[A]` — вложенные запросы и `WITH ... AS (...)` для читаемости. CTE — твой главный инструмент структурирования сложных запросов.

**S.6 Изменение данных, схема и связи** `[A]` — `INSERT`/`UPDATE`/`DELETE`; `CREATE TABLE`, типы колонок, `PRIMARY KEY`; **связи между таблицами** (1:1 / 1:N / N:M), внешние ключи (`FOREIGN KEY`) и **ER-диаграммы** — как таблицы хранятся и как связаны (основа Практикума перед JOIN); `CREATE VIEW` — **витрина данных** (сохранённый срез над `JOIN`ами для аналитики); как исследовать незнакомую схему (`information_schema`); зачем нужны индексы (на уровне идеи «ускоряют поиск»).

**S.7 PostgreSQL + pgvector** `[A]` ⭐ — расширение `vector`, тип `vector(n)`, операторы расстояния (`<=>` косинус, `<->` L2, `<#>` inner product), поиск ближайших через `ORDER BY ... LIMIT k`, индекс **HNSW**, и главное — **гибрид: векторный поиск + `WHERE`-фильтр по метаданным в одном запросе**. Здесь оживает косинусная близость из Этапа 1.1.3.

Минимальный пример того, как это выглядит (для интуиции):
```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE docs (
  id       bigserial PRIMARY KEY,
  category text,
  content  text,
  embedding vector(1536)          -- размерность под твою модель эмбеддингов
);

-- индекс под косинусную близость
CREATE INDEX ON docs USING hnsw (embedding vector_cosine_ops);

-- гибридный поиск: ближайшие по смыслу + фильтр по метаданным
SELECT id, content, embedding <=> :query_vec AS distance
FROM docs
WHERE category = 'faq'
ORDER BY embedding <=> :query_vec
LIMIT 5;
```
*Заметь:* это ровно тот самый «семантический поиск с фильтрацией», который ты руками делал на NumPy в Этапе 1 и 4.3 — только теперь на проде, в одном SQL-запросе.

**S.8 Text-to-SQL как агентный паттерн** `[A]` — модель генерирует SQL по вопросу пользователя, твой код исполняет, результат возвращается. Риски: **SQL-инъекции, права доступа, разрушительные запросы** — поэтому только read-only роль, белый список таблиц, лимиты, валидация. Связь с безопасностью инструментов из Этапа 4.6.6.

## Аналитический SQL — закрепление спринтов 1–2 Практикума

Эти темы ты **уже прошла** в Практикуме. Здесь они оформлены как чек-лист закрепления + привязка к роли AI Engineer, чтобы знания не выветрились и сразу работали на цель. Большинство — слой **[B]** (бонус для роли, но настоящий навык аналитики и частый вопрос на собеседовании); **ad-hoc и статистика — [A]** за прямую переносимость в анализ eval-логов.

**A.1 Множества и подзапросы** `[A]` — `UNION`/`INTERSECT`/`EXCEPT`, подзапросы, цепочки `CTE` *(спринт 1, тема 5)*.

**A.2 Категоризация и очистка** `[A]` — `CASE`, вычисляемые столбцы; **работа с дубликатами** — явные и неявные дубли, поиск и удаление (`DISTINCT`, `GROUP BY`, dedup через `ROW_NUMBER`) *(спринт 1, тема 6)*.

**A.3 Дата и время** `[A]` — типы, операторы, интервалы, группировка по периодам *(спринт 1, тема 7)*.

**A.4 Оконные функции: основы** `[B]` — `OVER`, агрегаты без схлопывания строк, `PARTITION BY`, `ORDER BY` *(спринт 2, тема 3)*.

**A.5 Ранжирование** `[B]` — `ROW_NUMBER`, `RANK`, `DENSE_RANK`, `NTILE` *(спринт 2, тема 4)*.

**A.6 Функции смещения** `[B]` — `LEAD`/`LAG`, `FIRST_VALUE`/`LAST_VALUE` *(спринт 2, тема 5)*.

**A.7 Описательная статистика в SQL** `[A]` ⭐ — меры центра и разброса, аналитические/перцентильные функции *(спринт 2, тема 6)*. Прямой мост к Этапу 1.4 — те же понятия с другой стороны.

**A.8 Ad-hoc методология** `[A]` ⭐ — декомпозиция бизнес-вопроса на CTE-шаги, итеративная сборка ответа *(спринт 2, тема 7)*. Это **тот же навык, что анализ логов eval** в Этапе 5: разбить вопрос на куски и собрать ответ.

> **Твои проекты Практикума** («статистика по клиентам», «Секреты Темнолесья») уже отрабатывают A.1–A.8 — отметь их в трекере как сделанные. А **позже, в Этапе 5**, перепиши анализ eval-логов своего RAG как ad-hoc-запрос с оконными функциями — это соединит SQL-навык с AI-инженерией и станет аккуратным пунктом портфолио.

## Проверка: две «Практики» (блоки) + один финальный проект (тема)
Проверок ровно два уровня — так задумано, без «докажи себе» и отдельных «контрольных» в тексте карты:
- **Каждый блок → своя «Практика»** — набор заданий, покрывающий все его подтемы; последняя задача блока = интегративная сборка (для S это гибридный pgvector-поиск).
- **Вся тема → один «Финальный проект»** — мини-проект, собирающий S+A вместе.

Три файла с разобранными задачами на общих seed-данных (логи LLM, eval, эмбеддинги), от простого к сложному:
- **[Практика S — рабочий SQL для LLM](../notes/sql/S-практика/README.md)** *(18 задач, S.1–S.8):* выборки, фильтры, **пропуски (`COALESCE`/`NULLIF`)**, джойны (вкл. `RIGHT`/`FULL`), CTE, изменение данных/схема, **витрины (`VIEW`)**, **pgvector** (гибридный векторный поиск ⭐) и безопасность **text-to-SQL**. Финал блока — собрать pgvector-поиск end-to-end. 📝 у пункта `S-практика` в блоке S трекера.
- **[Практика A — аналитический SQL](../notes/sql/практика/README.md)** *(13 задач, A.1–A.8):* множества (`UNION`/`EXCEPT`), `CASE`, дубликаты, даты, оконные функции, ранжирование, смещения, перцентили, ad-hoc. 📝 у пункта `практика` в блоке A трекера.
- **[Финальный проект темы — RAG-аналитика на SQL](../notes/sql/финал/README.md)** *(мини-проект, S+A):* полный аналитический отчёт по версиям промпта (окна + перцентили + ad-hoc) **+** RAG-retrieval с аналитикой на pgvector. Проходится последним. 📝 у пункта `финал` в блоке «🏁 Финал темы» трекера.

## Что НЕ нужно учить (чтобы не утонуть)
Глубокая оптимизация планов запросов, партиционирование, репликация, хранимые процедуры, транзакционные уровни изоляции в деталях, администрирование сервера. *(Оконные функции ты уже освоила в Практикуме — их не выкидываем, а держим как закрепление: см. A.4–A.6.)* Если что-то из «не нужного» встретится в конкретной вакансии — добавишь точечно уже в роли.

**NoSQL / MongoDB — на уровне «знать, что есть».** Документные БД встречаются в стеке (и Нетология, и часть вакансий упоминают их обзорно). Тебе достаточно понимать: это хранилище «документов» (JSON-подобных) без жёсткой схемы, удобно для неструктурированных данных; в мире LLM-приложений основная рабочая лошадка всё равно Postgres + pgvector. Отдельно учить Mongo сейчас не нужно — при необходимости разберёшься за вечер по официальной документации.
