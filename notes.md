## 1. Основы Python:  Циклы и их организация · Управление циклом. Операторы break, continue и pass

Чтобы **остановить выполнение цикла**, даже если остались итерации, используют оператор `break` (англ. «прерывать, прекращать»). Например, остановим цикл, когда значение переменной-итератора достигнет 5:
Скопировать код
PYTHON

```
for i in range(7):
    # Остановим цикл при значении переменной-итератора = 5    
    if i == 5:
        break
    print(i) 
```

Оператор `continue` (англ. «продолжать, возобновлять») позволяет **прервать текущую итерацию цикла** и перейти к следующей. В этом случае текущая итерация пропускается.
Например, пропустим итерацию цикла, когда значение переменной-итератора будет равно 2:
Скопировать код
PYTHON

```

for i in range(7):
    # Пропустим итерацию при значении переменной-итератора = 2
    if i == 2:
        continue
    # Остановим цикл при значении переменной-итератора = 5    
    if i == 5:
        break
    print(i) 
```

Этот оператор также поможет пропустить выполнение инструкций в **конкретном блоке программы**, если наступит определённое условие в цикле. Но в отличие от `continue`, который пропускает всю итерацию, `pass` просто **ничего не делает** в конкретном блоке, но цикл продолжает выполняться, включая инструкции после `pass`.
Разберём на примере, и если значение переменной-итератора будет равно 3, то в структуре программы запланируем блок с действиями, но оставим его без инструкций, а только укажем оператор `pass`:
Скопировать код
PYTHON

```
for i in range(7):
    # Пропустим блок программы, если значение переменной-итератора = 3
    if i == 3:
        # Пока нет инструкций для этого блока
        pass    
    # Пропустим итерацию при значении переменной-итератора = 2
    if i == 2:
        continue
    # Остановим цикл при значении переменной-итератора = 5    
    if i == 5:
        break
    print(i) 
```

Однако функция `enumerate()` уже возвращает индекс и значение всех элементов списка, что избавляет от необходимости ещё раз обращаться к элементу списка в теле цикла: `prices[i] - 100`.
Поэтому `enumerate()` предпочтительнее использовать при переборе и изменении элементов по их индексу вместо конструкции `range(len(some_list))`. Это сделает код проще для понимания и поможет избежать лишних обращений к списку или другому объекту.
А теперь небольшая тренировка.

## Кортежи

Списки изменяемы и этим очень удобны, однако иногда необходимо, чтобы созданную последовательность нельзя было изменить. В этом случае используют кортежи. Кортеж (англ. tuple) — это последовательность элементов, которые перечисляются через запятую в круглых скобках: `letters = ('A','B','C')`. Кортеж похож на список тем, что его элементы могут быть разного типа. После создания в кортеже нельзя добавлять, изменять или удалять элементы, поэтому они занимают меньше места в памяти компьютера.

ИИ не понимает, почему вы поссорились, какие эмоции вы испытываете, и не чувствует психологических нюансов. Он не оценивает, что в одной ситуации подарок может быть проявлением внимания, а в другой — выглядеть как попытка «купить прощение».
Для человека такие различия понятны: человек учитывает контекст, эмоции и подтекст ситуации. ИИ же работает иначе — для него это просто набор слов, из которых складывается шаблонный ответ.

####

В практике авторского права уже есть важные решения:

* Если произведение создано в основном нейросетью, без заметного творческого вклада человека, авторское право на него не предоставляется. Так трактует закон США.
* В деле Zarya of the Dawn регистрировали только часть, написанную человеком. Изображения, созданные ИИ, исключили из охраны, потому что автор не принимал участия в их создании.
* Крупные правообладатели уже подают иски против ИИ-сервисов. Они считают, что их материалы использовались без разрешения и нарушают авторские права.

## 25. Введение в ИИ · Как работают нейросети

При создании нейросети специалисту нужно описать общий вид такой формулы — из каких математических операций она будет состоять. Готовые наборы таких операций называются **слоями**. Они выполняются друг за другом и позволяют собрать нейронную сеть как конструктор.
Есть сотни разных слоёв для различных задач. Самые простые примеры — это **полносвязный слой (Linear)** или **нелинейный слой (ReLU)**. Полносвязный слой позволяет модели находить простые зависимости между признаками. Например, как сегодняшняя температура влияет на вероятность дождя или стаж разработчика — на его зарплату.
Нелинейный слой может находить более сложные закономерности. Например, если зарплата резко вырастает после достижения определённого стажа (скажем, после пяти лет), а до этого растёт медленно, линейный слой не сможет выразить такой скачок, а ReLU справится, отсекая значения по порогу.

Сначала входные данные передаются в линейные слои, затем в ReLU, а затем снова в линейные. Результат всех преобразований становится предсказанием нейросети. Такая нейросеть сможет предсказывать сумму двух простых зависимостей, например, сумму зарплаты и годового бонуса для разработчика.

Определённый порядок слоёв внутри сети называется **архитектурой нейросети**. Архитектуру можно считать своего рода чертежом, описывающим, какие слои использовать, в каком количестве и как их соединять между собой.
Архитектура определяет, как именно нейросеть будет обрабатывать информацию. От неё зависит, будет ли модель хорошо справляться с определённой задачей.

Данные бывают очень разными: картинки, тексты, звуки, таблицы — и для каждого типа подходят свои приёмы обработки. Например, чтобы распознать объект на фото, нейросети важно уметь «замечать» линии и углы. Поэтому для работы с изображениями часто применяют архитектуры, в которых много слоёв, способных хорошо «видеть» детали на картинках. Чтобы понять смысл предложения, важно уметь работать с порядком слов — с этим помогают другие архитектуры, которые лучше запоминают последовательность слов и выделяют важные фрагменты.
Поэтому универсальной архитектуры, которая бы идеально работала со всеми типами данных и задачами, пока нет. Для каждой задачи подбирается самая подходящая комбинация слоёв.
Некоторые архитектуры стали настолько известными, что получили собственные имена. Например:

* **Midjourney** — архитектура для генерации изображений. В ней используются слои, которые хорошо справляются с обработкой и созданием картинок.
* **GPT** — архитектура, созданная для работы с текстом. Такая сеть хорошо понимает смысл текста и может его продолжать.

Этапы обучения выглядят так:

1. Весам нейросети присваивают начальные значения. Это могут быть нули, единицы или случайные числа из нормального распределения.
2. В текущую формулу подставляют входные данные из примеров с известными ответами. Это позволяет определить **ошибку предсказания** — разницу между ответами, которые должны быть, и теми, что сейчас даёт нейросеть.
3. Далее происходит обновление весов. Благодаря тому, что известны вид формулы и ошибка предсказания, можно посчитать, как сильно и в какую сторону нужно изменить каждый коэффициент, чтобы сделать ошибку меньше.
4. После обновления формулы процесс возвращается к шагу 2.
5. Обучение продолжается, пока ошибка предсказания не станет достаточно маленькой — то есть пока нейросеть не начнёт давать ответы, близкие к правильным.

Поговорим про категорию нейронных сетей, оказавшую особое влияние на распространение ИИ-инструментов, — о **Больших Языковых Моделях** (англ. Large Language Models, LLM) и, в частности, о GPT.
Архитектура GPT используется во многих инструментах (например, ChatGPT или нейросеть Алиса (YandexGPT), но всех их объединяет общий принцип работы. Они могут принимать текст от пользователя и давать ответ, имитируя человеческую переписку. С их помощью можно решать самые разные задачи: генерировать код, находить объяснения интересующих вопросов или даже писать стихи.
Название архитектуры GPT расшифровывается как **Generative** **Pre-trained** **Transformer**, или **генеративный** **предобученный** **трансформер**.

Первое слово в аббревиатуре даёт понять, что нейросеть умеет генерировать новые данные. В качестве входных данных выступает текст запроса, который разбивается на фрагменты — **токены**. Дальше для простоты будем иметь в виду, что токен = слово.
Чтобы нейросеть могла подставить слова в свою формулу, их преобразуют в числовые векторы, которые хранят информацию об исходном семантическом смысле. Такие векторы называются **эмбеддингами.** Формула нейросети обрабатывает эмбеддинги и выдаёт ответ.

Здесь кроется главная хитрость: нейросеть не пишет весь текст сразу. Она даже не знает, о чём будет говорить дальше. Её задача — подбирать по одному слову, которое больше всего подходит для продолжения входящего текста. Например, для фразы «Ответ на твой…» самым логичным продолжением будет слово «вопрос».

Под «предсказанием следующего слова» подразумевается **расчёт вероятностей**. Во время обучения GPT тренируется не угадывать единственное правильное слово, а наиболее точно предсказывать вероятности для множества различных слов, которые могли бы продолжить текст.
Вероятности позволяют модели не только выбирать наиболее подходящее слово, но и создавать тексты с разнообразием и непредсказуемостью, близкими естественному человеческому языку.

Вы уже знакомы с концепцией трансформеров. Трансформер — это блок нейросети, который используется для работы с последовательностями данных. **Блок** — это удачная последовательность слоёв для какой-либо задачи.

Особенность трансформеров в том, что они не ограничиваются анализом слов по очереди. Вместо этого они взаимодействуют сразу со всеми словами, которые встречались раньше. Это возможно благодаря особому механизму, который назвали **механизмом внимания** (англ. self-attention). Он позволяет трансформеру анализировать разные части данных с разной степенью «внимательности» — то есть учитывать важные слова, которые могут повлиять на ответ, и игнорировать остальные.

Навыки, которым обучают нейросеть, идут от общего к частному: сначала нужно научить её понимать язык, а потом — правильно реагировать на команды и отвечать так, как этого ожидает человек.
Пока что мы рассказывали лишь об этапе обучения, на котором GPT учится генерировать текст. Этот шаг называется **предобучением**. На самом деле шагов три:

* предобучение;
* обучение на инструкциях;
* обучение с подкреплением.

#### **Шаг 1. Предобучение**

На этом этапе нейросеть изучает язык: она получает огромное количество текста для изучения и учится писать самостоятельно, постепенно улучшая результат. Этот процесс приводит к тому, что нейросеть может писать связный текст с соблюдением правил языка.
На этом этапе GPT умеет лишь выдавать продолжение текста (почти как Т9), не различая при этом команды и не умея общаться.

#### **Шаг 2. Обучение на инструкциях**

У этого этапа есть другое название — **контролируемая тонкая настройка** (англ. **Supervised** **Fine-Tuning**, SFT). Иногда вы будете встречать понятия «файнтюнинг» или «файнтюнить». Это значит настраивать нейросеть на выполнение конкретных задач — например, на поддержание диалога или написание кода.
На этом этапе GPT получает новые данные для обучения. Теперь это не просто текст. К каждому обучающему примеру прилагается правильный ответ — то, что в идеале должно получаться у нейросети. Если нужно сделать из нейросети хорошего голосового помощника, набор данных будет состоять из сообщений, которые она будет получать, и ожидаемых ответов.

#### **Шаг 3. Дообучение с подкреплением**

Во время обучения с подкреплением модель не видит верные ответы, но получает оценку — положительную, если её ответ хороший, и отрицательную, если он далёк от правильного. В этом случае задача нейросети — стараться отвечать так, чтобы получать только положительные оценки.
Этот этап позволяет GPT усовершенствовать свои результаты и научиться учитывать то, что хотят видеть пользователи в ответах. Например, ответ может быть верным фактически и грамматически, но содержать оскорбление или быть неполным. Благодаря дообучению GPT начинает общаться «по-человечески».

## 27. Практика решения ad hoc задач · Работа с декомпозированными частями ad hoc запроса

**Вычислим количество дней, которое прошло с последнего заказа.** Нужно вычислить разницу в днях между текущей датой и датой последнего заказа.
Скопировать код
SQL

```
CURRENT_DATE - MAX(o.created_at) AS days_since_last_order 
```

Напомним, что `CURRENT_DATE` в SQL возвращает текущую дату на момент выполнения запроса.

Задача «Нужны пользователи, которые сделали заказ в каждом из рекламных каналов». Ей соответствует запрос:
Скопировать код
SQL

```
SELECT o.user_id
FROM online_store.orders AS o
JOIN online_store.sessions AS s ON o.user_id = s.user_id
GROUP BY o.user_id
HAVING COUNT(DISTINCT s.channel) = (SELECT COUNT(DISTINCT channel) 
FROM online_store.sessions); 
```

Задача «Необходимы пользователи, которые делали заказы с нескольких устройств». Ей соответствует запрос:
Скопировать код
SQL

```
SELECT o.user_id
FROM online_store.orders AS o
JOIN online_store.sessions AS s ON o.user_id = s.user_id
GROUP BY o.user_id
HAVING COUNT(DISTINCT s.device) > 1; 
```

Задача «Выгрузить рекламный канал, который привлёк пользователей с наибольшей средней продолжительностью сессии». Ей соответствует запрос:
Скопировать код
SQL

```
SELECT s.channel
FROM online_store.sessions AS s
GROUP BY s.channel
ORDER BY AVG(s.session_duration) DESC
LIMIT 1; 
```

Задача «Определить категорию товаров с наибольшим количеством уникальных покупателей». Ей соответствует запрос:
Скопировать код
SQL

```
SELECT i.category
FROM tools_shop.order_x_item AS oxi
JOIN tools_shop.items AS i ON oxi.item_id = i.item_id
JOIN tools_shop.orders AS o ON oxi.invoice_id = o.order_id
GROUP BY i.category
ORDER BY COUNT(DISTINCT o.user_id) DESC
LIMIT 1; 
```

## 29. Описательная статистика. Аналитические функции · Аналитические функции

Чтобы вычислить процентиль в SQL, используем функцию `PERCENTILE_DISC(…) WITHIN GROUP (ORDER BY …)`. Разберём, за что отвечает каждая деталь этой команды:
Скопировать код
SQL

```
SELECT PERCENTILE_DISC(...) -- здесь указываем уровень процентиля
 WITHIN GROUP (ORDER BY ...)-- здесь указываем столбцы, по которым вычисляется процентиль
FROM ...; 
```

Так, 15-й процентиль в поле `session_duration` вычисляем с помощью следующего кода:
Скопировать код
SQL

```
SELECT PERCENTILE_DISC(0.15) -- здесь указываем уровень процентиля
 WITHIN GROUP (ORDER BY session_duration)-- здесь указываем столбцы, по которым вычисляется процентиль
FROM online_store.sessions; 
```

Скопировать код
SQL

```
 percentile_disc
-----------------
        5 
```

Получается, что 15% всех значений в поле `session_duration` — меньше 5.

SELECT PERCENTILE\_DISC(0.15) WITHIN GROUP (ORDER BY session\_duration) AS perc FROM online\_store.sessions GROUP BY region;

`PERCENTILE_CONT()` использует два соседних реально имеющихся процентиля из нашего набора данных. В том же примере выше — 15-й процентиль в 1, 2, 3, 4, 5 — функция `PERCENTILE_CONT()` обнаружила бы в наборе данных 10-й и 20-й процентили и с их помощью вычислила бы 15-й процентиль — 1.5

Как видим, результат не изменился. Возможно, это связано с тем, что в нашей таблице много строк, а для таких наборов данных `PERCENTILE_DISC()` обычно работает точно. Но если в исходном наборе данных будет меньше 100 строк, как в примере с 1, 2, 3, 4, 5, то `PERCENTILE_DISC()` может выдать неточный результат. Поэтому всегда выбираем между `PERCENTILE_DISC()` и `PERCENTILE_CONT()`, учитывая в основном объём набора данных.

Стандартное отклонение вычисляется в SQL с помощью функции `STDDEV(...)`. Вычислим стандартное отклонение поля `session_duration`:
Скопировать код
SQL

```
SELECT STDDEV(session_duration) AS stand_dev
FROM online_store.sessions; 
```

Также стандартное отклонение можно вычислять не только для всего столбца, но и как оконную функцию. Давайте вычислим стандартное отклонение для каждого региона в отдельности и впишем результат как новый столбец в таблице:
Скопировать код
SQL

```
SELECT user_id,
       region,
       STDDEV(session_duration) OVER (PARTITION BY region) AS stand_dev
FROM online_store.sessions; 
```

## Размах

Самая простая и базовая величина, определяющая разброс некоторого набора данных, — <strong>размах</strong>. Он определяется как разность между максимальным и минимальным значениями набора данных. Например, для набора значений -1, 2, 5, 10 размах будет равен 11 (10 - (-1)).

36+25+16+9+16+64​≈27.67
Эта величина является средним квадратом отклонений от среднего набора данных, но называют её **дисперсией**

Добавим третий столбец в нашу таблицу и впишем квадрат отклонения:

| Значение | Отклонение от среднего | Квадрат отклонения от среднего |
| -------- | ---------------------- | ------------------------------ |
| 1 | 1 - 7 = -6 | (-6) \* (-6) = 36 |
| 2 | 2 - 7 = -5 | (-5) \* (-5) = 25 |
| 3 | 3 - 7 = -4 | (-4) \* (-4) = 16 |
| 10 | 10 - 7 = 3 | 3 \* 3 = 9 |
| 11 | 11 - 7 = 4 | 4 \* 4 = 16 |
| 15 | 15 - 7 = 8 | 8 \* 8 = 64 |

И после этого считаем среднее арифметическое не просто отклонений, а их квадратов. Тогда получаем:
36+25+16+9+16+646≈27.67\\dfrac{36 + 25 + 16 + 9 + 16 + 64}{6} \\approx 27.67636+25+16+9+16+64​≈27.67
Эта величина является средним квадратом отклонений от среднего набора данных, но называют её **дисперсией**. Формулу этой величины и её свойства мы обсудим в следующих уроках.
Только чаще всего пользуются не дисперсией, а квадратным корнем из дисперсии — тем самым **стандартным отклонением**. Для нашего примера стандартное отклонение будет равно:
27.67≈5.26\\sqrt{27.67} \\approx 5.2627.67​≈5.26

Стандартное отклонение здесь позволяет описать весь набор значений одной фразой: «Набор значений со средним, равным 7, и стандартным отклонением, примерно равным 5». То есть в основном наши значения распределены где-то в диапазоне от 2 (7 - 5) до 12 (7 + 5), что является правдой. Именно так, благодаря мерам центральной тенденции, и разброса, появляется возможность ёмко и коротко описывать довольно большие и сложные наборы значений.

Ещё стандартное отклонение иногда называют **стандартной ошибкой**. Почему так происходит, легче понять с помощью следующего примера. Представим, что спортсменка стреляет в мишень, центр которой — в значении 0. Но никто не застрахован от ошибок, поэтому она попадает не в 0, а вокруг этого значения.

Знать про меры разброса важно так же, как понимать, когда и какую величину стоит использовать. Размах может подойти в ситуациях, когда все значения сгруппированы достаточно близко и нет одиноких больших значений. Стандартное отклонение же — довольно универсальный инструмент: его можно использовать практически с любыми данными.

## 31. Описательная статистика. Аналитические функции · Меры центральной тенденции

Любая характеристика, которая описывает тем или иным способом центр набора данных, называется **мерой центральной тенденции**

## Мода

Самая базовая мера центральной тенденции — <strong>мода</strong>. Она показывает, какое значение встречается чаще всего.

И финальным шагом выбираем именно то значение, которое встречается больше всего раз. Целых 3 раза нам встретилось значение 2, оно и будет модой этого набора.

Но что делать, если в наборе данных каждое значение встречается только один раз? Чтобы с этим разобраться, возьмём такое количество заказов пользователями за год: 9, 1, 2, 7, 3, 6, 5. Наша задача — описать центр этого набора значений, чтобы прогнозировать количество заказов каждого пользователя и выручку с каждого пользователя в следующем году.
![Изображена шкала с количеством заказов от 0 до 10. Здесь отмечены значения из набора данных: 1, 2, 3, 5, 6, 7, 9.](https://pictures.s3.yandex.net/resources/image_1724082950.png)
Для такого набора данных вычислять моду бесполезно, потому что каждое значение встречается такое же количество раз, как и остальные. Определить центральную тенденцию нам поможет <strong>медиана</strong>. Медиана показывает центральное значение в упорядоченном наборе данных. Чтобы её вычислить, нужно выполнить три шага:

1. Отсортировать набор значений по возрастанию.
2. Посмотреть, сколько в наборе значений: чётное или нечётное количество.
3. Если количество нечётное, то центральный элемент в наборе и будет медианой. Если чётное — медианой может быть любое значение между левым и правым центральными, но чаще берут среднее этих двух центральных значений.

Разберём другой набор данных: 1, 2, 5, 8, 100, где нужно учесть каждое значение. Он представляет собой количество тысяч рублей, которое потратили пользователи на заказы в этом месяце. Наша задача — вычислить центр этого набора данных, чтобы узнать приблизительный доход на следующий месяц.
![Изображена шкала с количеством заказов от 0 до 100. Здесь отмечены значения из набора данных — 1, 2, 5, 8, 100.](https://pictures.s3.yandex.net/resources/image_1724082973.png)
Мода здесь нам не поможет, так как частота всех значений одинакова. Медиана — тоже, потому что она никак не учтёт большое значение 100 справа. Зато нас выручит <strong>среднее арифметическое</strong>: с ним вы уже знакомы. Среднее арифметическое использует в вычислении каждое значение и потому отлично подходит, когда мода и медиана недостаточно описывают весь набор значений.

Термин «процентиль» (percentile) появился в статистике и происходит от английского слова, которое состоит из двух частей: - *percent* («процент») — от латинского <em>per centum</em>, что значит «на сотню»; - *ile* — суффикс, который часто используется в статистических и математических терминах.

* **Первый квартиль** — значение, меньше или равно которого — 25% набора данных.
* **Второй квартиль** — значение, меньше или равно которого — 50% набора данных. Можно заметить, что второй квартиль является медианой, так как медиана делит набор данных на две равные по объёму части: показывает значение, меньше или равно которому — 50% набора данных.
* **Третий квартиль** — значение, меньше или равно которого — 75% набора данных.

Стоит учесть, что процентили и квартили не относятся к мерам центральной тенденции, так как не описывают центр. Тем не менее они считаются описательными статистиками.

## 32. Описательная статистика. Аналитические функции · Описательные статистики

Определить, является ли величина количественной или категориальной, можно с помощью простого теста: если два значения величины можно сложить друг с другом — это количественная величина, если нельзя — категориальная.

## 35. Оконные ранжирующие функции · Функции RANK() и DENSE_RANK()

Функция `RANK()` (от англ. «ранг, категория, классифицировать») последовательно ранжирует строки, согласно выбранному порядку сортировки данных, но в отличие от `ROW_NUMBER()` в случае нахождения одинаковых строк возвращает им одинаковый ранг с пропуском последующего значения ранга.

В примере выше функция `ROW_NUMBER()` ранжировала строки последовательно, при этом участники с одинаковым результатом получили разные призовые места. А функция `RANK()` ранжировала участников по спортивному принципу — участники с одинаковым результатом получили одинаковое призовое место. Затем был пропуск рангов, и следующий участник получил ранг согласно его порядковому номеру в таблице, отсортированной по результатам забега.

## Функция `DENSE_RANK()`

Функция `DENSE_RANK()` (от англ. «плотный ранг») также позволяет ранжировать объекты по определённому значению или значениям, однако в отличие от `RANK()` ранги нумеруются последовательно, без пропусков в номере ранга:

```
-- Подсчёт количества заказов для каждого клиента: 
WITH top_users AS (
    SELECT DISTINCT
        user_id,
        -- Оконная функция, подсчитывающая количество заказов для каждого клиента:
        COUNT(event_dt) OVER(PARTITION BY user_id) AS total_orders
    FROM online_store.orders
    ORDER BY total_orders DESC
)
SELECT 
    -- Оконная функция, ранжирующая строки по порядку:
    ROW_NUMBER() OVER(ORDER BY total_orders DESC),
    *,
    -- Оконная функция, ранжирующая строки по порядку с учётом значения в строке: 
    RANK() OVER(ORDER BY total_orders DESC),
    -- Оконная функция, ранжирующая строки по порядку с учётом значения в строке
    -- без пропусков в интервале рангов: 
    DENSE_RANK() OVER(ORDER BY total_orders DESC)
FROM top_users
LIMIT 15;

## 36. Оконные ранжирующие функции · Ранжирующие оконные функции. Функция ROW_NUMBER()

Функция `ROW_NUMBER()` присвоит каждому заказу порядковый номер вне зависимости от значений в других столбцах. Результат войдёт в отдельное поле, назовём его — `row_num`. Обратите внимание, что `ROW_NUMBER()` не требует аргумента:
Скопировать код
SQL

```
SELECT 
    *,
    -- Оконная функция, добавляющая ранг каждой строке:
    ROW_NUMBER() OVER() AS row_num
FROM online_store.orders
LIMIT 10; 
```

. Если результат запроса будет отсортирован по условию в секции `ORDER BY` основного запроса, то функция `ROW_NUMBER()` всё равно пронумерует записи, исходя из их исходного порядка в таблице, то есть не будет учитывать результат сортировки в секции `ORDER BY`.

Скопировать код
SQL

```
SELECT 
    *,
    -- Оконная функция, добавляющая ранг каждой строке с учётом сортировки данных:
    ROW_NUMBER() OVER(ORDER BY revenue DESC) AS row_num
FROM online_store.orders
ORDER BY row_num
LIMIT 10; 
```

Если указать ORDER BY внутри оператора OVER (), то в каждой строке подсчёт будет идти от начала окна до текущей строки:

Представим, что у нас есть такая таблица `online_store.orders` с заказами интернет‑магазина:

| -------- | ------- | ------- |

| 2023-01-02 | 102 | 200 |
| 2023-01-03 | 103 | 150 |
| 2023-01-04 | 104 | 300 |

Скопировать код
SQL

SELECT

    user_id,
    revenue,

|  | `PARTITION BY` | `GROUP BY` |
| --- | ------------ | -------- |
| Что делает | Разбивает строки на окна и при этом в результате сохраняет все строки | Группирует строки и при этом уменьшает количество строк в результате |

#### **Использование PARTITION BY с датой**

`PARTITION BY` можно использовать, чтобы разделять данные по дате. Это поможет вычислять ежемесячные, еженедельные или ежедневные суммы и средние значения:
Скопировать код
SQL

```
SELECT event_dt,
       revenue,
       SUM(revenue) OVER (PARTITION BY DATE_TRUNC('day', event_dt)) AS daily_revenue
FROM online_store.orders; 
```

| event\_dt | revenue | daily\_revenue |
| -------- | ------- | ------------- |
| 2020-06-14 | 4.42 | 3228.98 |
| 2020-06-14 | 2.23 | 3228.98 |
| 2020-06-14 | 8.12 | 3228.98 |
| 2020-06-14 | 8.05 | 3228.98 |
| … | … | … |

В этом примере `SUM(revenue) OVER (PARTITION BY DATE_TRUNC('day', event_dt))` разбивает строки на окна по дням и для каждого окна вычисляет сумму поля `revenue`. Можно увидеть, сколько составляет общая выручка за каждый день.

## 41. Оконные функции. Агрегирующие функции · Агрегирующие оконные функции

У обычных агрегирующих функций, которые используют с оператором `GROUP BY`, ограниченные возможности: они группируют данные и возвращают одно значение для каждой группы, исключая при этом остальные строки. Из-за этого становится невозможно выполнять сложные задачи — расчёт кумулятивных сумм или среднего значения для каждой строки по всей таблице.

Оператор `OVER ()` — ключевой элемент оконных функций в SQL. Он определяет, в пределах какого «окна» строк будет выполняться вычисление. Если к агрегирующей функции добавить `OVER ()`, функция превратится в оконную и добавит результат к каждой строке в качестве нового столбца.

Когда оператор используется без параметров, «окном» для вычисления агрегирующей функции будет вся таблица:
Скопировать код
SQL

```
SELECT столбец1,
       столбец2,
       <агрегирующая_функция>(column_name) OVER () AS новое_значение
FROM table; 
```

С оконными агрегирующими функциями можно сочетать функции для округления, например `ROUND()`:
Скопировать код
SQL

```
SELECT user_id,
       session_duration,
       ROUND(AVG(session_duration) OVER (), 2) AS avg_session_duration
FROM online_store.sessions; 
```

Важно также знать, что результат работы оконной функции нельзя отфильтровать в запросе с помощью `WHERE`, потому что оконные функции выполняются после фильтрации и группировки. Это означает, что фильтрация происходит на этапе, когда оконные функции ещё не были вычислены.

```
SELECT user_id,
       session_duration,
       avg_session_duration
FROM
  (SELECT user_id,
          session_duration,
          AVG(session_duration) OVER () AS avg_session_duration
   FROM online_store.sessions) AS subquery
WHERE session_duration > avg_session_duration; 
```

Что тут происходит:

1. Подзапрос вычисляет среднюю продолжительность сессий для всех строк таблицы с помощью оконной функции `AVG(session_duration) OVER ()`. Результат вычисления сохраняется в столбце `avg_session_duration`.
2. Внешний запрос фильтрует строки, оставляя только те записи, в которых продолжительность сессии `session_duration` больше средней продолжительности сессий `avg_session_duration`.

В запросе оконная функция `AVG(session_duration) OVER ()` вычисляет среднюю продолжительность сессий для всех строк таблицы и добавляет результат в новый столбец `avg_session_duration`. Подзапрос нужен, чтобы вычислить среднее значение и добавить его к каждой строке таблицы. Фильтрация же выполняется во внешнем запросе, в котором сохраняются только те строки, в которых продолжительность сессии больше среднего значения.

## 42. Оконные функции. Агрегирующие функции · Оконные функции

**Оконными** (англ. window functions) называют функции, которые выполняют вычисления по набору строк, которые определённым образом связаны. В отличие от обычных агрегирующих функций, которые группируют строки и возвращают один результат для каждой группы, оконные функции сохраняют строки и добавляют вычисленное значение к каждой строке в результате.

Оконные функции выполняют вычисления для набора записей, объединённых по какому-либо признаку. Такой объединённый набор называют окном — отсюда и название функций.

Правильный ответ
Агрегирующие функции используются для группировки строк и возвращают одно значение для каждой группы, а оконные функции сохраняют порядок строк и добавляют вычисляемое значение к каждой строке

Одно из распространённых применений — расчёт кумулятивных сумм и средних значений. Например, в таблице `orders` из схемы `online_store` с помощью оконных функций можно посчитать общую выручку по дням для каждого пользователя. Так вы сможете увидеть, как значение выручки накапливалось с течением времени.
Кумулятивная сумма для пользователя рассчитывается так: выручка за первый день прибавляется к выручке за второй день, затем к этой сумме добавляется выручка за третий день и так далее. Такие расчёты помогут понять, как накапливается общая выручка по мере совершения заказов:
![Оконные функции можно использовать для расчёта кумулятивной суммы. Так можно увидеть, как значение накапливается с течением времени.](https://pictures.s3.yandex.net/resources/image_1723479842.png)
Сразу скажем, что аналогичного результата можно добиться и без оконных функций — с помощью подзапросов. Такой SQL-запрос рассчитает кумулятивную выручку для каждого пользователя:
Скопировать код
SQL

```
SELECT o1.user_id,
       o1.event_dt,
       o1.revenue,
    (
    -- Вложенный запрос для расчёта кумулятивной выручки
        SELECT SUM(o2.revenue)
        FROM orders AS o2
    -- Условие накопления: включаем заказы до текущей даты
        WHERE o2.user_id = o1.user_id AND o2.event_dt <= o1.event_dt
    ) AS cumulative_revenue -- Кумулятивная (накопительная) выручка
FROM orders AS o1
ORDER BY o1.user_id,
         o1.event_dt; 
```

Однако с помощью оконных функций такие расчёты будут выполняться гораздо проще. В следующих уроках мы это покажем и познакомим вас с синтаксисом.

#### **Агрегирующие оконные функции**

Агрегирующие оконные функции выполняют вычисления над набором строк в окне и возвращают одно значение для каждой строки в этом окне. Помните, в отличие от обычных агрегирующих функций, они сохраняют порядок строк и добавляют вычисляемое значение к каждой строке. Это позволяет считать кумулятивные суммы или средние значения, не уменьшая количество строк в результате. Ещё один пример использования таких функций — вычисление среднего рейтинга для каждого пользователя по годам.

## 43. Знакомство с базой данных · Создание витрины данных

**Витрина данных** (англ. Data Mart) — это предварительно подготовленная таблица со всеми необходимыми данными для обучения модели или решения аналитической задачи. Она содержит структурированные и осмысленные данные, готовые к использованию без дополнительных преобразований и сложных SQL-запросов.

На этапе исследовательского анализа данных достаточно объединять таблицы `SELECT`-запросами любой длины. Однако при создании моделей это уже неудобно: алгоритмам машинного обучения нужны аккуратно организованные данные с релевантной для бизнес-задачи информацией. Витрина решает эту проблему, объединяя разрозненные данные в таблицу с продуманной структурой.

При этом, разумеется, витрины нужны далеко не всегда. Их создание целесообразно только в ряде случаев:

* Для модели нужны агрегированные показатели, рассчитанные одинаково для всех наблюдений в любой момент.
* Задача повторяется. Если вы запускаете одни и те же расчёты каждую неделю, лучше один раз собрать витрину и обновлять её.
* Вы работаете с большим объёмом данных. Благодаря витринам не придётся каждый раз ждать выполнения сложного запроса — у вас уже будет готовая таблица со всем необходимым.
* Над проектом работает команда. Витрина упрощает согласование рабочего процесса — все опираются на одни и те же данные.
* Нужно обеспечить воспроизводимость. Например, модель дообучают раз в месяц, и важно, чтобы данные всегда считались одинаково.

Для ряда же других задач витрины будут избыточны:

* Вы выполняете разовую задачу. Под неё нет смысла собирать отдельную таблицу — достаточно одного запроса.
* Нужен «свежий» результат. Например, если вы анализируете данные по звонкам в реальном времени. Витрина формируется с задержкой, поэтому может устареть.
* Набор показателей постоянно меняется. Если вы экспериментируете с новыми данными и их структура ещё не устоялась, то создавать витрину явно рано.
* Вы работаете с малым объёмом данных. Малые выборки не требуют оптимизации через витрины.

Чтобы использовать результаты для построения модели, нужно сохранить их в виде витрины. В SQL для создания новой таблицы на основе `SELECT`-запроса используют команду `CREATE TABLE <имя_витрины> AS`:
Скопировать код
SQL

```
-- Команда для сохранения вывода SELECT-запроса в витрину:
CREATE TABLE <имя_витрины> AS
SELECT ... 
```

После запуска этого кода вы не просто получаете таблицу на экране, как при обычном `SELECT`-запросе, а создаёте новую таблицу, которая теперь хранится в базе данных. В неё сразу записываются все строки, которые возвращает ваш запрос.
Чтобы убедиться в том, что таблица существует, выведем все таблицы в схеме:
Скопировать код
SQL

```
-- Выведем названия всех таблиц схемы online_store:
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'online_store'; 
```

Если витрина больше не нужна, вы можете удалить её командой `DROP TABLE`:
Скопировать код
SQL

```
-- Удаление витрины user_features_daily:
DROP TABLE online_store.user_features_daily; 
```

Здравствуйте!
Отличный вопрос — он как раз помогает разобраться в сути витрин данных.
Короткий ответ: <strong>нет</strong>, данные в витрине не обновятся автоматически.
Разберём подробнее, почему так происходит:

1. **Как создаётся витрина.** Когда вы создаёте витрину (например, с помощью `CREATE VIEW` или сохраняете результат `SELECT` в отдельную таблицу), система выполняет запрос один раз и формирует набор данных — сохраняет его в новом объекте БД.
2. **Связь с исходными данными.** Витрина «помнит» только сам SQL‑запрос, а не прямую ссылку на исходные строки. То есть в ней лежат конкретные значения, которые были получены в момент создания или последнего обновления.
3. **Что происходит при изменении данных.** Если в исходной таблице изменится имя пользователя, это никак не затронет уже готовую витрину — в ней останутся старые значения.

***

**Какие есть варианты обновления витрины:**

* **Пересоздать витрину** — заново выполнить запрос, который её формировал. Это можно сделать вручную или по расписанию.
* **Настроить автоматическое обновление** — некоторые СУБД и ETL‑инструменты (например, Apache Airflow, dbt) позволяют запускать обновление витрин по триггеру (время, событие и т. п.).
* **Использовать материализованные представления** (`MATERIALIZED VIEW`) — в некоторых СУБД они могут автоматически синхронизироваться с изменениями в исходных таблицах (но это зависит от настроек и возможностей системы).

***

Получается, витрины — это скорее «снимок» данных на определённый момент времени, а не постоянно обновляемое зеркало. Поэтому важно продумать процесс их регулярного обновления в вашем проекте.

## 44. Знакомство с базой данных · Изучаем БД самостоятельно

Помимо данных БД PostgreSQL содержит набор таблиц и представлений, содержащих метаданные БД или системную информацию о самой базе данных — о схемах, таблицах, зависимостях между ними, о пользователях и ролях, а также другую системную информацию. Такие таблицы и представления содержат в названии префикс `pg_` и хранятся в системной схеме PostgreSQL `pg_catalog`.

По умолчанию все таблицы, если им не указана определённая схема, помещаются в схему `public`. Эта схема входит в каждую новую базу данных PostgreSQL, и чтобы обратиться к ней, не нужно использовать её имя.

Перед началом работы с данными стоит проверить, что вы обращаетесь к таблице с необходимыми данными в нужной схеме. Чтобы выгрузить имена доступных схем в базе данных, следует выполнить такой запрос:
Скопировать код

```
-- Выведем имена всех схем базы данных:
SELECT schema_name

Чтобы получить названия всех таблиц, можно воспользоваться запросом:
Скопировать код

## 45. Линейная алгебра:  Векторы и тригонометрия · Понятие вектора

**Вектор** — это направленный отрезок, для которого указаны его начало и конец.

У любого вектора есть две основные характеристики: длина и направление. Вектор можно как угодно перемещать в пространстве, главное, чтобы длина и направление не менялись. Если же изменить хоть один из этих параметров, получится новый вектор.

Координаты точки — это набор чисел, с помощью которого можно однозначно восстановить положение точки на плоскости или в пространстве.

По сути такой вектор является точкой. Значит, каждая точка плоскости — это нулевой вектор, все координаты которого равны нулю. Например, OO‾=(0, 0).\\overline{OO}=(0{,}\\ 0).OO=(0, 0).

С точки зрения геометрии два вектора называют <strong>равными</strong>, если они одинаково направлены и их длины равны.

## 46. Работа с датой и временем · Операторы для работы с датой и временем

#### <strong>Функция `EXTRACT`**

Функция `EXTRACT` позволяет выделить часть даты или времени, её отдельные элементы, например номер месяца, недели, дня и тому подобное.
Например, вам нужно выделить месяц из даты типа `timestamp`. Дата выглядит следующим образом: `1983-05-15`. В таких случаях можно применить функцию `EXTRACT`:
Скопировать код
SQL

```
SELECT EXTRACT(MONTH FROM CAST('1983-05-15' AS timestamp)); 
```

Отрезок времени может быть представлен следующими значениями:

* `CENTURY` — век;
* `DAY` — день;
* `DOY` (от англ. day of the year) — день года, выраженный числом от 1 до 365 или 366, если год високосный;
* `DOW` (от англ. day of the week) — день недели, выраженный числом от 0 до 6, где понедельник — 1, воскресенье — 0;
* `ISODOW` (от англ. day of the week и ISO 8601) — день недели, выраженный числом от 1 до 7, где понедельник — 1, воскресенье — 7;
* `HOUR` — час;
* `MILLISECOND` — миллисекунда;
* `MINUTE` — минута;
* `MONTH` — месяц;
* `SECOND` — секунда;
* `QUARTER` — квартал;
* `WEEK` — неделя в году;
* `YEAR` — год.

А если необходимо определить, например, интервал действия определённой цены в таблице `dishes_prices`, код может выглядеть так:

## 47. Работа с датой и временем · Типы данных для даты и времени

Для ответа на этот вопрос вспомним, для чего вообще существуют типы данных. Говоря простым языком, когда мы назначаем данным тип, то мы как бы объясняем базе данных, какие операции теперь можно с этими данными производить. Например, если мы говорим, что какие-то данные числовые, значит, с ними можно производить математические операции или использовать агрегирующие функции. Если данные строчные, то в них можно искать отдельные символы, слова, объединять или разделять строки. Их тоже можно агрегировать, но ограниченно: например, можно подсчитать количество строк, а вот сумму подсчитать не получится — строки складывать в SQL нельзя. Та же ситуация и с типом даты и времени.

СУБД PostgreSQL позволяет работать с самыми разными вариантами типов данных, подбирать тип под конкретные задачи, создавать пользовательские типы. Все типы данных не охватить за нескольких уроков, да и необходимости такой нет. Вы научитесь пользоваться самыми распространёнными.

Один из самых распространённых типов — содержит дату и время. Чаще всего дату и время указывают в формате ISO: `'YYYY-MM-DDTHH:MM:SS'`.

Например, нам необходимо выяснить, сколько времени прошло между `'2010-06-30 16:05'` и `'2010-07-08 13:08'`. Самым логичным действием будет вычитание одной даты из другой.
Скопировать код
SQL

```
SELECT '2010-07-08 13:08' - '2010-06-30 16:05'; 
```

И при такой записи система выдаст ошибку — строки вычитать нельзя. Но если привести к нужному типу:
Скопировать код
SQL

```
SELECT '2010-07-08 13:08'::timestamp - '2010-06-30 16:05'::timestamp; 
```

В PostgreSQL типов `timestamp` два:

* `timestamp with time zone` включает данные о часовом поясе;
* `timestamp without time zone` без данных о часовом поясе.
    Часовые пояса указывают по стандарту UTC: `'2004-10-19 10:23:54+02'`. В часовом поясе UTC+2 находится, например, Калининград. Сравните со значением типа `timestamp without time zone`: `'2004-10-19 10:23:54'`. По стандарту SQL обозначение `timestamp` по умолчанию подразумевает тип `timestamp without time zone`. Для обозначения типа `timestamp with time zone` в PostgreSQL используют короткую форму `timestamptz`.

#### <strong>Тип `date`**

По сути, тип `date` — аналог типа `timestamp` по методам преобразования и свойствам, но в него входит только дата. Дату можно задавать в любом формате, например October 19, 2004, 2004-Oct-19 или даже 20041019.

Будьте осторожны! Как только вы преобразуете данные к типу `date`, вы потеряете информацию о времени.

#### <strong>Тип `time`**

Тип `time` — аналог типа `date`, но оставляет информацию только о времени. Такой тип может понадобиться, если нужно только время, без учёта даты. Например, это может быть расписание автобуса или запуск ежедневного события в определённое время:

#### <strong>Тип `interval`**

В предыдущих примерах мы уже вычитали из одной даты другую. При этом результат был не просто числом, а указывался с временным отрезком — днями. Такой тип данных, получаемый при разности двух дат, называется `interval`. Он служит, как следует из названия, для работы с временными интервалами.
Записывается он в свободной текстовой форме: `'2 weeks 1 day 36 minutes'` или `'1 year 5 days 1 hour'` и так далее.
Выполним простую задачу при помощи интервала — прибавим к дате `'20-03-2024'` пять месяцев:
Скопировать код
SQL

```
SELECT CAST('20-03-2024' AS date) + CAST('5 months' AS interval) 
```

Использование интервалов полезно при формировании отчётности по времени. Например, такие задачи, как «отобразить число пользователей, которые зарегистрировались на пятый день после маркетинговой акции» или «итоговая выручка с каждого покупателя через месяц после его регистрации», удобно решать, используя интервалы.

У интервалов есть одна математическая особенность — их можно складывать и вычитать между собой. Например, если сложить интервал в 10 лет и интервал в 5 лет и 12 часов, получится интервал в 15 лет и 12 часов. Так же работает и вычитание интервалов. Кроме того, можно назначить псевдоним для таких столбцов или результатов:
Скопировать код
SQL

```
SELECT '7 years'::interval - '6 years 11 months'::interval AS inter_val; 
```

| inter\_val |
| --------- |
| 30 days, 0:00:00 |

**Unix-время** (англ. Unix time, также POSIX-время) — система описания моментов во времени, принятая в Unix и других POSIX-совместимых операционных системах. Определяется как количество секунд, прошедших с полуночи (00:00:00 UTC) 1 января 1970 года. Этот момент называют «эпохой Unix» (англ. Unix epoch).

Работать с таким форматом сложно, поскольку он фактически не отображает привычную дату. Но выход есть! В PostgreSQL есть оператор, который преобразует время Unix в уже знакомый `timestamp`.

SQL

## 49. Операции множеств и подзапросы · СТЕ, или Обобщённые табличные выражения

В SQL существует механизм, который позволяет вынести подзапрос за пределы основного запроса. Для этого используют **обобщённые табличные выражения, или** **CTE** (от англ. common table expressions). Они помогают структурировать подзапросы и сохранить их результат для повторного использования.

К CTE можно обращаться после `FROM` или оператора `JOIN`, но нельзя после `WHERE`. В PostgreSQL есть такое ограничение.

В CTE можно вычислить среднее значение какого-нибудь показателя и использовать его несколько раз в запросе. Предположим, вы посчитали среднее количество клиентов в обобщённом табличном выражении и использовали его для расчёта среднего чека и среднего количества заказов на одного клиента. При использовании подзапросов пришлось бы для каждого расчёта указывать один и тот же подзапрос.

## 50. Операции множеств и подзапросы · Подзапросы

Обратите внимание, что подзапросу в секции `WHERE` не нужно задавать псевдоним. Наоборот, если его назначить, то запрос вызовет ошибку, так как псевдонимы в секции `WHERE` не используются. Важно также проверить, чтобы результатом подзапроса в секции `WHERE` было одно поле. В таком поле может быть даже одно значение, и такой подзапрос будет называться <strong>скалярным</strong>. Если поле хранит множество значений, подзапрос будет <strong>многострочным</strong>.

##

Обратите внимание: в подзапросе было условие, что при расчёте количества заказов `user_id` клиента должен соответствовать `user_id` из таблицы `users`. Это необходимо для любого подзапроса в секции `SELECT` — в нём должно быть условие на соответствие значения поля из подзапроса значению поля во внешнем запросе.

Обратите внимание: как и в случае с подзапросом в `WHERE`, подзапрос в `SELECT` должен содержать в качестве результата только одно поле. Если полей будет несколько, произойдёт ошибка.

Указывать псевдоним для подзапроса в `SELECT` необязательно. Если указать псевдоним, новое поле получит именно это название, несмотря на то, как поле названо в подзапросе.

На практике часто бывает, что одну и ту же задачу можно решить разными способами. Нет чётких инструкций, когда следует использовать подзапросы, а когда — решать задачу с помощью присоединения таблиц. Однако стоит учитывать несколько моментов:

* Большие подзапросы увеличат длину кода и сделают запрос сложнее для восприятия другими аналитиками. Однако подзапросы уменьшат объём данных, который вам понадобится обработать на следующих этапах.
* Если вы присоединяете большие по объёму данные, в результате получите ещё более объёмные таблицы, а это увеличит нагрузку на базу данных. Преимущество подзапросов в том, что они позволят отсечь неиспользуемую информацию до решения основной задачи.

## 53. Присоединение таблиц · Операторы LEFT JOIN и RIGHT JOIN

Может показаться, что тип `RIGHT JOIN` вообще не нужен. Однако он может быть полезен по нескольким причинам:

* **Сделает запрос читаемым и понятным.** Например, в случае, если логика бизнес-процесса предполагает работу с правой таблицей как с основной. Представьте, что вы работаете с запросами, которые были давно написаны к таблицам `users` и `orders`, и хотите их дополнить, а не переписывать. Допустим, вам нужно узнать, какие заказы были сделаны, даже если для них в системе нет соответствующих пользователей. Если основной таблицей считать правую, лучше использовать `RIGHT JOIN`. Таблицу с пользователями вы укажете левой, то есть после `FROM`, а таблицу с заказами — правой, то есть после оператора `RIGHT JOIN`.
* **Сохранит согласованность и структуру запроса.** Например, вы работаете с таблицами `orders`, `users` и `cities` и хотите получить информацию о заказах, включая пользователей и города. Не для всех пользователей и городов будут заказы, поэтому тут может помочь `RIGHT JOIN`. Вы можете использовать оператор, чтобы сначала объединить `orders` и `users`, а затем добавить `cities`. При этом запрос будет логичным и понятно структурированным.
* **Позволит соблюсти стандарты.** В некоторых командах или организациях может быть принято использовать `RIGHT JOIN` в определённых ситуациях. Это может быть связано с корпоративными стандартами или предпочтениями разработчиков.

Также сообщим, что иногда в документации или интернет-ресурсах вы можете встретить названия `LEFT OUTER JOIN`, `RIGHT OUTER JOIN` и `FULL OUTER JOIN`. `OUTER JOIN` — это расширение стандартного типа `INNER JOIN`. Оно включает в себя три типа: `LEFT OUTER JOIN`, `RIGHT OUTER JOIN` и `FULL OUTER JOIN`.

## 54. Присоединение таблиц · Присоединение таблиц

При использовании `JOIN` может возникнуть ошибка `reference is ambiguous`. Такое происходит, если поле, которое вы используете в запросе, присутствует в обеих таблицах, и SQL не может определить, из какой таблицы взять данные. Приведём пример:
Скопировать код
SQL

```
SELECT city_id, -- Поле присутствует в таблицах users и cities
       first_name,
       last_name
FROM rest_orders.users
JOIN rest_orders.cities ON rest_orders.users.city_id = rest_orders.cities.city_id; 
```

SQL не знает, из какой таблицы взять поле `city_id`, ведь оно присутствует в обеих таблицах и сообщает об ошибке:
Скопировать код
SQL

```
ERROR:  column reference "city_id" is ambiguous
LINE 2:     city_id 
```

Чтобы её избежать, нужно явно указывать таблицы:
Скопировать код
SQL

```
SELECT users.city_id AS user_city_id,
       cities.city_id AS city_city_id,
       users.first_name,
       users.last_name
FROM rest_orders.users
JOIN rest_orders.cities ON rest_orders.users.city_id = rest_orders.cities.city_id; 
```

```
JOIN rest_orders.cities AS c ON u.city_id = c.city_id; 
```

В этом запросе `u` и `c` — это псевдонимы таблиц `users` и `cities` соответственно. В условии явно указывается, что присоединение должно происходить по столбцам `city_id` в таблице `users` и `city_id` в таблице `cities`.
Вместо этого можно использовать ключевое слово `USING`, чтобы сделать запрос более читаемым:
Скопировать код
SQL

```
SELECT u.user_id,
       u.first_name,
       u.last_name,
       c.city_name,
       u.gender,
       u.birth_date,
       u.registration_date
FROM rest_orders.users AS u
JOIN rest_orders.cities AS c USING (city_id); 
```

## 57. Работа с пропущенными значениями и дубликатами · Работа с пропусками

Допустим, вы создаёте базу данных на основе адресов пользователей. Вы учли все возможные варианты адресов для регистрации, поэтому пользователь может заполнить следующее: `Город`, `Район`, `Строение`, `Ближайшее метро`, `Дом`, `Корпус`, `Этаж`, `Квартира` и так далее. В некоторых случаях пользователи не смогут заполнить все поля, если, например, человек живёт в городе без метро. Тогда в базе данных в поле `Ближайшее метро` соответствующей строки будет записано значение `NULL`. Однако не вся ответственность за пропуски лежит на пользователях: причиной `NULL` могло быть плохое соединение, из-за которого в базу загрузились не все ответы.

Ещё одна причина появления `NULL` в базе — создание новых полей. По умолчанию в новом поле для всех существующих записей будет присвоено значение `NULL`.

Одна из тактик работы с пропусками — оставить их без изменений и анализировать остальные значения. Всё потому, что заменить `NULL` не так просто, и выбирать значение на замену нужно с умом. Если выбрать неподходящее значение, можно прийти к неверным выводам.
Если же удалить все строки с `NULL`, можно исказить данные и в результате всё равно получить ошибку анализа. У каждого варианта работы с `NULL` свои недостатки, поэтому на практике `NULL` часто и вовсе оставляют без изменен

Для поиска значений без `NULL` создан специальный оператор: `WHERE … IS NOT NULL`. А для поиска `NULL` -значений используется `WHERE … IS NULL`

`COUNT(*)` показывает общее количество строк в таблице. Наличие `NULL` на результат не повлияет, ведь количество строк не изменится. В свою очередь, операции, которым в качестве аргумента передают конкретный столбец, например, `AVG`, `SUM`, `COUNT` и аналогичные им, не будут учитывать значения `NULL` при вычислениях.

`COUNT(*)` показывает общее количество строк в таблице. Наличие `NULL` на результат не повлияет, ведь количество строк не изменится. В свою очередь, операции, которым в качестве аргумента передают конкретный столбец, например, `AVG`, `SUM`, `COUNT` и аналогичные им, не будут учитывать значения `NULL` при вычислениях.
Предполложим, в поле `price` хранятся значения `(4, 4, 4, 4, NULL)`. Результат расчётов функции `AVG(price)` будет равен 4, `SUM(price)` будет равен 16, а `COUNT(price)` — 4. Из-за таких особенностей стоит аккуратно и внимательно применять агрегирующие функции в таблицах со значениями `NULL`.

Вы уже знаете, что представляют собой значения `NULL` и как с ними работать. Самое время обсудить, чем они могут быть полезны. В прошлом уроке мы говорили, что пропуски не являются чем-то плохим, а несут информацию о качестве данных. Перед тем как удалить пропуски из базы, важно посмотреть, сколько их, и оценить качество данных.
Проверим пропуски в поле `discount`. Для этого используем `COUNT(*)` вместе с оператором `WHERE`:
Скопировать код
SQL

```
SELECT COUNT(*)
FROM rest_orders.orders 
WHERE discount IS NULL; 
```

| count |
| ----- |
| 22104 |

В результате получим, что в 22104 строках обнаружены значения `NULL`. Чтобы найти количество строк без пропусков, можно просто использовать `COUNT(discount)`, ведь `COUNT(discount)` и так проигнорирует `NULL`-значения:
Скопировать код
SQL

```
SELECT COUNT(discount)
FROM rest_orders.orders; 
```

```
SELECT COUNT(discount) / COUNT(*)
FROM rest_orders.orders; 
```

В результате такого запроса получим 0. Догадались почему? Всё дело в том, что `COUNT(discount)` и `COUNT(*)` — целые числа. В результате их деления мы получим только целую часть ответа. Аналогично, если разделить `5` на `2`, получится не `2.5`, а целая часть `2.5` — `2`:
Скопировать код
SQL

```
 SELECT 5 / 2; 
```

| ?column? |
| -------- |
| 2 |

Чтобы в результате получить вещественное число, приведём одно из выражений к типу для вещественных чисел: `numeric`, `real` или `double precision`. На примере с `5` и `2` убедимся, что это работает:
Скопировать код
SQL

```
 SELECT 5 / CAST(2 AS real); 
```

Если вы всё-таки обнаружили дубликаты и пропуски, важно обратить внимание, в каких полях они находятся.

## 59. Связи между таблицами · ER-диаграммы

Деление таблицы на две в реляционной базе данных при связи «один к одному» может быть целесообразным в нескольких случаях:

1. **Улучшение читаемости и структурированности данных:** если таблица содержит много столбцов, которые можно логически разделить на две группы, разделение может сделать структуру данных более понятной.
2. **Оптимизация производительности:** иногда разделение таблицы может ускорить выполнение запросов, особенно если одна из частей таблицы используется гораздо чаще другой.
3. **Обеспечение целостности данных:** в некоторых случаях разделение таблицы может помочь лучше контролировать права доступа к различным частям данных или обеспечить более строгие правила целостности.
4. **Нормализация базы данных:** разделение таблицы может быть частью процесса нормализации, который направлен на уменьшение избыточности данных и улучшение их структурной целостности.

**ER-диаграммы** (от англ. entity–relationship diagram — «диаграмма „сущность — связь“»)

**Схема базы данных** — это набор связанных между собой таблиц. Схема похожа на контейнер, в котором сгруппированы связанные объекты базы: таблицы и другие элементы. Схемы помогают структурировать базу и сгруппировать таблицы по общему признаку.

Схемы в базе данных помогают:

* **Структурировать данные и их взаимосвязи.** Если у базы логичная структура, работать с ней становится проще. Например, таблицы, связанные с пользователями, могут быть сгруппированы в одной схеме, а таблицы, связанные с заказами, — в другой.
* **Управлять доступом.** Схемы позволяют устанавливать права доступа к разным объектам базы. Например, можно дать доступ к одной схеме только отделу аналитики, а к другой — отделу маркетинга.
* **Сделать анализ проще.** Благодаря схемам становится проще понять структуру базы данных и писать запросы. Например, если все таблицы, связанные с продуктами, находятся в схеме `products`, то это позволит быстрее найти нужные данные.
* **Изолировать объекты.** Схемы позволяют изолировать объекты, чтобы избежать конфликтов имён и улучшить управление. Например, если в базе данных две таблицы с `orders` (одна для интернет-магазина, другая для физического), их можно разместить в разных схемах — `online_store.orders` и `physical_store.orders`.

Схемы помогут избежать конфликта имён в случае, если в базе существует несколько таблиц с одинаковыми названиями, например `orders`:
Скопировать код
SQL

```
-- Извлечение данных из таблицы orders в схеме online_store
SELECT * 
FROM online_store.orders;

-- Извлечение данных из таблицы orders в схеме physical_store
SELECT * 
FROM physical_store.orders; 
```

При написании SQL-запросов рекомендуем указывать схему перед именем таблицы, чтобы избежать неоднозначности и ошибок. Особенно если вы работаете с большой базой данных со множеством схем и объектов. Однако схему можно не указывать, если в базе данных одна схема — `public`, которая автоматически назначается в PostgreSQL.

## 60. Связи между таблицами · Связи между таблицами

* **Естественный первичный ключ** (англ. primary key, PK) — это поле в таблице базы данных, которое идентифицирует каждую запись, например, номер паспорта.
* **Суррогатный первичный ключ** — это случайные или специально созданные номера, которые система присваивает каждой записи, чтобы её идентифицировать. Такие ключи не меняются со временем и не зависят от каких-либо изменений в данных.
* **Составной первичный ключ** — это ключ, который состоит из двух или более столбцов. Вместе эти столбцы образуют уникальный идентификатор для каждой строки в таблице. Составной ключ используется, когда один столбец не может однозначно идентифицировать запись.

#### **Связь «многие ко многим»**

Такая связь в базах данных обозначает ситуацию, когда одна запись в одной таблице может быть связана с несколькими записями в другой таблице, и наоборот. Чтобы реализовать такую связь, обычно используют промежуточную таблицу. Она содержит внешние ключи, которые указывают на первичные ключи двух связанных таблиц. Промежуточную таблицу ещё называют таблицей-связью, или join-таблицей (англ. join — «соединять»).
