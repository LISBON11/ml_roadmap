# ЭТАП 5 — Инференс, деплой, продакшн и выход на роль

*Часть 6 (финальная) из дорожной карты Frontend → AI Engineer.*

**Слой: [A]** · **Длительность: ~6–8 недель** при 20–25 ч/нед.

**Зачем этот этап.** Здесь флагманский проект из Этапа 4 превращается в **задеплоенный продукт с живой ссылкой**, а абстрактная цель «сменить профессию» — в измеримый чек-лист. Рынок 2026 хочет не «знатоков промптов», а инженеров, которые **доставляют надёжные AI-продукты**: employers want people who can build useful LLM applications, not just call a model API. Твой фронтенд здесь — не довесок, а ниша: full-stack AI-инженеров с нормальным UX мало.

### Где запускать (Mac vs облако)
- **Инференс через hosted API (основной режим портфолио):** Mac, GPU не нужна. 🍎
- **Локальные модели:** Ollama на Mac MPS — для приватных/офлайн-экспериментов с малыми моделями.
- **Серьёзный self-hosted сервинг (vLLM):** нужна GPU → облако. ☁️ Для портфолио это опционально — достаточно понимать концепции.
- **Деплой:** фронтенд — Vercel/Netlify; бэкенд — Modal/Railway/Fly.io/Render; демки — Hugging Face Spaces.

## Блок 5.1 — Инференс и оптимизация `[A]`

**Зачем:** понимать стоимость/скорость генерации и варианты запуска моделей — частый предмет собеседований («ты когда-нибудь укладывался в бюджет?»).

**5.1.1 Как считается инференс** — токены/сек, латентность, throughput, KV-кэш, prompt caching. 🎯 *Докажи себе:* измерь латентность первого токена и полную для одного запроса.

**5.1.2 Квантизация** — GGUF, AWQ, GPTQ, bitsandbytes; компромисс размер/скорость/качество. 🎯 *Докажи себе:* запусти одну открытую модель в двух квантизациях через Ollama и сравни скорость/качество.

**5.1.3 Локальный инференс** — Ollama на Mac (MPS), когда локально оправдано (приватность, оффлайн, цена). 🎯 *Докажи себе:* подними локальную модель в Ollama и сделай к ней API-запрос.

**5.1.4 Hosted inference и сервинг** — провайдерские API, vLLM/TGI для самостоятельного сервинга, speculative decoding (обзор). 🎯 *Докажи себе:* объясни, когда выгоднее hosted API, а когда self-hosted.

**5.1.5 Оптимизация стоимости** ⭐ — выбор модели под задачу, роутер моделей (мощная → средняя → дешёвая), кэширование, ограничение длины. 🎯 *Докажи себе:* посчитай стоимость 10-ходового диалога (8K вход / 1K выход на ход) и предложи, как её срезать вдвое.

**Ресурсы:** Ollama docs (EN); DeepLearning.AI «Efficient LLM Inference with vLLM» (EN); статьи провайдеров про prompt caching и роутинг моделей (EN, первоисточники). — *проверено на июнь 2026.*

**Контрольное:** возьми свой RAG из Этапа 4, посчитай стоимость одного типового запроса и снизь её (модель/кэш/длина) без потери качества по eval.

## Блок 5.2 — Деплой full-stack AI-приложения `[A]` ⭐ (твоё преимущество)

**Зачем:** живой задеплоенный проект с UI — главное доказательство «умею довести до продакшена». Hiring-менеджеры screen for production deployment evidence first — not Kaggle medals, not Coursera certificates.

**5.2.1 Бэкенд для LLM-приложения** — FastAPI (стандарт в вакансиях), асинхронность, стриминг ответов клиенту. 🎯 *Докажи себе:* подними FastAPI-эндпоинт, стримящий ответ модели.

**5.2.2 Фронтенд** ⏭️ ⭐ — твой React-стек: стриминг, generative UI, отображение источников; Vercel AI SDK. 🎯 *Докажи себе:* выведи стриминговый ответ в React-компонент с индикацией источников.

**5.2.3 Связка и развёртывание** — фронт на Vercel, бэк на Modal/Railway/Fly, переменные окружения, секреты ключей. 🎯 *Докажи себе:* задеплой минимальный full-stack AI-чат с публичной ссылкой.

**5.2.4 Управление ключами и лимитами** — серверное хранение ключей (никогда на клиенте), rate limiting, защита от перерасхода. 🎯 *Докажи себе:* убедись, что API-ключ не утекает в браузер, и добавь лимит запросов.

**5.2.5 Демо для портфолио** — Hugging Face Spaces / Streamlit / Gradio как быстрый способ показать. 🎯 *Докажи себе:* подними демо одного проекта на HF Spaces.

**5.2.6 Контейнеризация (Docker)** ⏭️ — упаковка приложения в образ, `Dockerfile`, переменные окружения, запуск контейнера; кратко про деплой контейнера в облако (включая Yandex Cloud для РФ-рынка). Тебе как инженеру с опытом контейнеризации это пойдёт быстро, но в вакансиях AI Engineer Docker — стандартный инструмент, поэтому держим явной строкой. 🎯 *Докажи себе:* собери `Dockerfile` для своего FastAPI-бэкенда и запусти контейнер локально.

**Ресурсы:** FastAPI docs (EN); Vercel AI SDK docs `ai-sdk.dev` (EN); docs хостингов (Vercel, Modal, Railway, HF Spaces) — EN; официальная документация Docker. — *проверено на июнь 2026.*

**Контрольное по блоку 5.2:** задеплой флагманский проект Этапа 4 с публичной ссылкой, серверным хранением ключей и работающим стримингом в UI.

## Блок 5.3 — Наблюдаемость, стоимость, безопасность `[A]` ⭐

**Зачем:** это водораздел «демо vs продакшн». В 2026 — обязательные навыки: production observability, cost optimization, safety/guardrails.

**5.3.1 Трейсинг и наблюдаемость** ⭐ — логирование вызовов, трейсы агентов; LangSmith, Langfuse, Helicone; OpenTelemetry как стандарт. 🎯 *Докажи себе:* подключи Langfuse/LangSmith и посмотри трейс одного RAG-запроса.

**5.3.2 Мониторинг качества в проде** — онлайн-eval на трейсах (LLM-as-judge), алерты на регрессии, отслеживание task success / latency / cost-per-task. 🎯 *Докажи себе:* настрой метрику «cost-per-task» и порог алерта.

**5.3.3 Безопасность LLM-приложений** ⭐ — OWASP Top 10 for LLM Applications; **prompt injection**, особенно непрямой через извлечённые документы («новый XSS»). 🎯 *Докажи себе:* проведи red-team своего RAG — найди 1 вектор инъекции через документ и закрой его.

**5.3.4 Guardrails** — фильтрация входа/выхода (Llama Guard, NeMo Guardrails), ограничение области инструментов, структурированный вывод как ограничение. 🎯 *Докажи себе:* добавь выходной фильтр, блокирующий нежелательный контент.

**5.3.5 Excessive Agency (LLM06)** ⭐ — агент совершает необратимые действия вне замысла; митигации: минимальные привилегии инструментов, человек-в-цикле, kill switch. 🎯 *Докажи себе:* объясни на своём агенте, где он мог бы «перестараться», и как ты это ограничил.

**Ресурсы:** OWASP Top 10 for LLM Applications — *первоисточник, EN, бесплатно* (`owasp.org`); docs Langfuse/LangSmith/Helicone (EN); DeepLearning.AI «Red Teaming LLM Applications» (EN). — *проверено на июнь 2026.*

**Контрольное по блоку 5.3:** добавь к флагманскому проекту трейсинг (Langfuse/LangSmith), один guardrail и краткий отчёт red-team с тремя векторами инъекций и митигациями.

## Блок 5.4 — Основы MLOps по необходимости `[A]`

**Зачем:** минимум инженерной гигиены для AI-приложений; глубокий классический MLOps для LLM-роли не нужен.

**5.4.1 Версионирование промптов и конфигов** ⭐ — промпт в гите с changelog, к каждому изменению привязан eval-результат. 🎯 *Докажи себе:* заведи версионирование системного промпта с привязкой eval-числа к каждой версии.

**5.4.2 Эксперименты** — трекинг прогонов (Weights & Biases / простые таблицы), воспроизводимость. 🎯 *Докажи себе:* залогируй 3 эксперимента с промптами и сравни их.

**5.4.3 CI/CD для AI-приложений** ⏭️ — автотесты + eval как «ship gate» в пайплайне (тебе CI/CD знаком). 🎯 *Докажи себе:* добавь шаг в CI, прогоняющий eval-набор и падающий при регрессии.

**5.4.4 Данные и приватность (обзор)** — что нельзя слать в API, PII, требования (для EU — про AI Act полезно знать). 🎯 *Докажи себе:* перечисли, какие данные твоего приложения нельзя отправлять в сторонний API.

**Ресурсы:** Weights & Biases docs/курсы (EN); общие практики из вакансий (versioning, eval-as-gate). — *проверено на июнь 2026.*

**Контрольное:** сделай так, чтобы изменение промпта в твоём проекте автоматически прогоняло eval и не мёржилось при падении метрики.

## Блок 5.5 — Портфолио: 3–4 проекта `[A]`

**Зачем:** портфолио важнее сертификатов. Идеи специально подобраны под твоё преимущество — **фронтенд + full-stack**.

**Проект 1 — Chat-with-your-docs (флагман).** RAG-приложение: загрузка документов → векторный поиск с re-ranking → ответы с цитатами → eval-дашборд. *Что показывает:* RAG, эмбеддинги, векторная БД, eval, деплой. *Фронтенд-угол:* стриминг, подсветка источников, приятный UX. Это сборка из вех Этапов 4–5.

**Проект 2 — Агентный ассистент с generative UI.** Агент с tool use (поиск, вызовы API), который **динамически рендерит UI** (графики, формы, карточки) через Vercel AI SDK. *Что показывает:* агенты, function calling, надёжность, безопасность инструментов. *Фронтенд-угол:* generative UI — то, что почти никто из ML-щиков не умеет.

**Проект 3 — Prompt/Eval-студия.** Веб-инструмент сравнения промптов/моделей с метриками (RAGAS, LLM-судья), трекингом стоимости и латентности, A/B. *Что показывает:* **eval-грамотность — сильнейший сигнал реального опыта** (eval design is the single best signal of real LLM experience) + cost-мышление. *Фронтенд-угол:* дашборды и визуализация — твоя стихия.

**Проект 4 — MCP-интеграция.** Свой MCP-сервер (обёртка над реальным API/данными) + хост-UI, либо подключение готовых серверов. *Что показывает:* MCP — горячий навык 2026 (его прямо просят в вакансиях), плюс твой опыт интеграций.

🎯 *Докажи себе (на каждый проект):* живая публичная ссылка + README с архитектурой, метриками eval и разделом «чему научился».

**Минимум для подачи:** Проект 1 (флагман) обязательно задеплоен, плюс ещё 1–2 поменьше. Все — с живыми ссылками и кодом на GitHub.

## Блок 5.6 — Чек-лист «готова к найму» + подготовка к собеседованию `[A]`

### Чек-лист компетенций AI Engineer (сверен с вакансиями 2026)
Рынок 2026 заменил резюме «LangChain + Pinecone + ChatGPT-обёртка» на набор продакшн-навыков. Отмечай галочками — это и есть измеримая цель «сменить профессию».

**Фундамент**
- [ ] Python (включая async) на рабочем уровне
- [ ] Понимаю трансформеры/attention, токены, эмбеддинги, сэмплирование (Этап 3 A-срез)
- [ ] Базовые метрики и статистика для оценки (Этапы 1.4, 2.2)

**Ядро LLM-приложений**
- [ ] LLM API ≥2 провайдеров (например, OpenAI + Anthropic), структурированный вывод, стриминг
- [ ] Промт-инжиниринг + context engineering; промпты версионирую как код
- [ ] RAG end-to-end: чанкинг, эмбеддинги, векторная БД, гибридный поиск, re-ranking, цитаты
- [ ] Агенты: function calling/tool use, ReAct, надёжность, безопасность инструментов
- [ ] MCP: понимаю архитектуру, могу написать свой сервер
- [ ] **Eval: проектирую тест-наборы, метрики RAG, LLM-as-judge** (сильнейший сигнал)

**Продакшн**
- [ ] Деплой full-stack приложения с публичной ссылкой
- [ ] Наблюдаемость (трейсинг: LangSmith/Langfuse/Helicone)
- [ ] Cost-оптимизация: умею оценить и снизить стоимость
- [ ] Безопасность: OWASP LLM Top 10, prompt injection, guardrails
- [ ] Версионирование промптов + eval как ship-gate

**Твоё преимущество (выделяет тебя)**
- [ ] Full-stack: React-фронт + FastAPI-бэк, generative UI, стриминг
- [ ] 3–4 задеплоенных проекта с живыми ссылками и измеримыми результатами

**Готова к подаче, когда:** отмечены все пункты «Ядро» + ≥4 из «Продакшн» + ≥3 задеплоенных проекта (включая флагман) + можешь пройти 6 типов вопросов ниже.

### Подготовка к собеседованию (6 типовых блоков)
1. **Eval-история** ⭐ — расскажи про eval, который ты спроектировал (главный фильтр «реально строил vs смотрел ютуб»). Подготовь конкретный пример из Проекта 1/3.
2. **Cost-моделирование** — оцени стоимость 10-ходового диалога (8K вход / 1K выход на ход); спроектируй роутер моделей (мощная→средняя→дешёвая). Это underrated in interviews but massively over-indexed on once in the role.
3. **Безопасность** — назови 3 категории OWASP LLM Top 10 без подсказки; объясни LLM06 Excessive Agency и митигации (минимальные привилегии, человек-в-цикле, kill switch).
4. **RAG-архитектура** — спроектируй RAG для корпоративного поиска: чанкинг, гибридный поиск, re-ranking, eval, борьба с галлюцинациями.
5. **Решение fine-tune vs RAG vs prompt** — обоснуй выбор по стоимости/латентности/точности.
6. **Red-team** — покажи 3 вектора инъекции в данном системном промпте и митигации за 15 минут.

Плюс обычное: системный дизайн LLM-приложения и поведенческое интервью (история смены профессии — превращай фронтенд-опыт в плюс).

**Ресурсы:** OWASP LLM Top 10 (EN); разделы eval/агенты Prompt Engineering Guide (EN); разборы вопросов в гайдах по найму 2026 (EN). — *проверено на июнь 2026.*
