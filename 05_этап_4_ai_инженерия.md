## Блок 4.1 — LLM API `[A]`

**Зачем:** базовый навык — разговаривать с моделью из кода. Для тебя API освоится мгновенно, важны нюансы токенов и структурированного вывода.

**4.1.1 Модели и провайдеры** — закрытые (GPT, Claude, Gemini) vs открытые (Llama, Mistral, Qwen, DeepSeek); как выбирать под задачу/цену/приватность. 🎯 *Докажи себе:* сделай один и тот же запрос к двум провайдерам и сравни ответы и стоимость.

> **Заметка про российские LLM.** Для рынка РФ актуальны YandexGPT и GigaChat (Сбер) — их часто требуют из-за приватности данных и импортозамещения; локальные открытые модели гоняются через Ollama (см. 5.1.3). Но ты целишься в EU/Amsterdam — там в разы важнее свободно владеть OpenAI / Anthropic / Gemini и их экосистемой. Вывод: освой паттерн «работа с LLM через API» так, чтобы он **переносился на любого провайдера**, а конкретные российские модели трогай точечно, только если конкретная вакансия их требует.

**4.1.2 Chat/Messages API** ⏭️ — роли `system`/`user`/`assistant`, передача истории диалога, многоходовость. 🎯 *Докажи себе:* собери мини-чат, сохраняющий контекст последних 5 сообщений.

**4.1.3 Токены и стоимость** ⭐ — input/output токены, ценообразование, контекстное окно, подсчёт стоимости запроса. 🎯 *Докажи себе:* посчитай стоимость одного длинного запроса в токенах и деньгах до отправки.

**4.1.4 Параметры генерации** — temperature, top_p, max_tokens, stop (связь с Этапом 3.7.4). 🎯 *Докажи себе:* прогони один промпт с temperature 0 и 1.2, сравни стабильность.

**4.1.5 Стриминг** ⏭️ — потоковая выдача токенов (твой фронтенд-навык SSE/streams). 🎯 *Докажи себе:* выведи ответ модели в терминал по мере генерации.

**4.1.6 Структурированный вывод** ⭐ — JSON mode, схемы (tool/function), валидация через Pydantic. 🎯 *Докажи себе:* заставь модель вернуть валидный JSON по заданной схеме и распарси его в типизированный объект.

**4.1.7 Мультимодальность (обзор)** — vision (картинки на вход), аудио. 🎯 *Докажи себе:* отправь изображение и попроси описать его.

**Ресурсы:** официальные docs провайдеров (EN); DeepLearning.AI «Building Systems with the ChatGPT API» (EN); DeepLearning.AI «Getting Structured LLM Output» (EN).

**Контрольное:** напиши тонкую обёртку-клиент над одним провайдером со стримингом, подсчётом токенов/стоимости и структурированным выводом в Pydantic-модель.

## Блок 4.2 — Промт-инжиниринг `[A]`

**Зачем:** дешёвый рычаг качества; в 2026 расширился до «context engineering» — что и в каком порядке класть в окно.

**4.2.1 Системные промпты и роль** — задание поведения, персоны, ограничений. 🎯 *Докажи себе:* одним системным промптом стабильно меняй тон ответов.

**4.2.2 Zero-shot и few-shot** — обучение примерами в промпте. 🎯 *Докажи себе:* подними качество классификации, добавив 3 примера.

**4.2.3 Chain-of-thought и reasoning** — пошаговое рассуждение, reasoning-модели, когда они оправданы (дороже/медленнее). 🎯 *Докажи себе:* реши логическую задачу с «думай по шагам» и без — сравни.

**4.2.4 Структурирование промпта** — инструкции в начале/конце, XML/JSON-разметка, разделители. 🎯 *Докажи себе:* оберни длинный контекст в XML-теги и попроси приоритет «критической» секции.

**4.2.5 Prompt chaining и декомпозиция** — разбиение сложной задачи на цепочку вызовов. 🎯 *Докажи себе:* реши задачу «классифицировать → обработать → проверить» тремя связанными вызовами.

**4.2.6 Context engineering** ⭐ — что положить в окно, в каком порядке, как держать свежим; recency/primacy bias, «потеря в середине». 🎯 *Докажи себе:* покажи, как перестановка важной инструкции в начало/конец меняет ответ.

**4.2.7 Итеративная отладка** — промпт как эмпирическая наука: менять по одному, мерить. 🎯 *Докажи себе:* заведи табличку «версия промпта → результат на 5 примерах».

**Ресурсы:** Prompt Engineering Guide (EN); DeepLearning.AI «ChatGPT Prompt Engineering for Developers» (EN); гайды по промптингу в docs провайдеров (например, гайд Anthropic по промпт-инжинирингу) — EN, первоисточники.

**Контрольное:** возьми задачу извлечения структурированных данных из «грязного» текста и доведи промпт до стабильного результата на 10 примерах; зафиксируй версии и метрику.

## Блок 4.3 — Эмбеддинги и семантический поиск `[A]`

**Зачем:** основа RAG и любого «умного поиска»; здесь оживает косинусная близость из Этапа 1.1.3.

**4.3.1 Эмбеддинг-модели** — что это, размерность вектора, закрытые (OpenAI embeddings) vs открытые (sentence-transformers), мультиязычные. 🎯 *Докажи себе:* получи эмбеддинг одной фразы и посмотри размерность вектора.

**4.3.2 Семантическая близость** ⭐ — косинус на практике (Этап 1.1.3), ранжирование по близости. 🎯 *Докажи себе:* найди в наборе из 50 фраз 3 ближайшие к запросу.

**4.3.3 Чанкинг** — разбиение документов на куски, размер/перекрытие, чанкинг по смыслу. 🎯 *Докажи себе:* разбей длинный документ двумя способами и сравни, какой даёт более осмысленные куски.

**4.3.4 Виды поиска** — семантический vs ключевой (BM25) vs гибридный. 🎯 *Докажи себе:* придумай запрос, где ключевой поиск выигрывает у семантического, и наоборот.

**4.3.5 Re-ranking** — переупорядочивание кандидатов отдельной моделью для точности. 🎯 *Докажи себе:* объясни, зачем нужен re-ranker, если уже есть векторный поиск.

**Ресурсы:** Hugging Face (sentence-transformers, модели эмбеддингов) — EN; DeepLearning.AI «Embeddings» / «Vector Databases» (EN); docs провайдеров по embeddings (EN).

**Контрольное:** построй семантический поиск по своему набору документов (чанкинг → эмбеддинги → косинус-топ-K) **без векторной БД**, на чистом NumPy/коде — как развитие вехи Этапа 1.

## Блок 4.4 — Векторные базы данных `[A]`

**Зачем:** хранить и быстро искать миллионы векторов. Для тебя это «просто ещё одна БД» — концептуально пойдёт быстро.

**4.4.1 Зачем нужны и ANN-поиск** — приближённый поиск ближайших соседей, почему не перебор. 🎯 *Докажи себе:* объясни, почему точный перебор не масштабируется на миллион векторов.

**4.4.2 Обзор решений** — pgvector (расширение Postgres — ближе всего тебе как фуллстеку), Qdrant, Chroma, Pinecone, Weaviate, FAISS. 🎯 *Докажи себе:* подними pgvector локально и вставь несколько векторов.

**4.4.3 Индексы и метрики** — HNSW, метрики расстояния (косинус, L2, dot). 🎯 *Докажи себе:* объясни, что настраивает HNSW и какой компромис́с он даёт.

**4.4.4 Метаданные и фильтрация** — гибрид «вектор + фильтр по полям». 🎯 *Докажи себе:* сделай поиск, ограниченный фильтром по метаданным (например, по дате).

**4.4.5 Выбор БД** — managed vs self-hosted, когда хватает pgvector. 🎯 *Докажи себе:* в 3 строках обоснуй выбор БД для прототипа vs продакшена.

**Ресурсы:** официальные docs (pgvector на GitHub, Qdrant, Chroma) — EN; DeepLearning.AI «Vector Databases» (EN). — *проверено на июнь 2026.*

**Контрольное:** перенеси семантический поиск из блока 4.3 на pgvector или Qdrant с фильтрацией по метаданным.

## Блок 4.5 — RAG (Retrieval-Augmented Generation) `[A]` ⭐

**Зачем:** ключевой паттерн «дать модели свежие/приватные знания без дообучения». Самый частый прод-кейс и почти обязательный пет-проект.

**4.5.1 Архитектура RAG** — индексация (загрузка → чанки → эмбеддинги → БД) + извлечение → дополнение промпта → генерация. 🎯 *Докажи себе:* нарисуй полную схему RAG со стрелками данных.

**4.5.2 Naive RAG и его проблемы** — нерелевантные чанки, галлюцинации, шум. 🎯 *Докажи себе:* собери простейший RAG и найди пример, где он отвечает мимо.

**4.5.3 Продвинутые техники** — query rewriting, гибридный поиск, re-ranking, contextual retrieval. 🎯 *Докажи себе:* добавь re-ranking и покажи рост релевантности.

**4.5.4 Цитирование и борьба с галлюцинациями** — ссылки на источники, «отвечай только по контексту». 🎯 *Докажи себе:* заставь RAG возвращать ответ со ссылками на конкретные чанки.

**4.5.5 Границы RAG** — «потеря в середине», размер контекста, когда RAG vs длинный контекст vs fine-tuning. 🎯 *Докажи себе:* объясни, в каком случае ты выберешь fine-tuning вместо RAG.

**Ресурсы:** Prompt Engineering Guide — раздел RAG (EN); DeepLearning.AI «Building and Evaluating Advanced RAG» и «LangChain: Chat with Your Data» (EN); блоги-первоисточники по RAG (например, Anthropic «Contextual Retrieval») — EN.

**Контрольное по блоку 4.5:** собери RAG над набором своих документов с цитированием источников и измеримым качеством (передаётся в блок 4.8 на eval).

## Блок 4.6 — Агенты и tool use / function calling `[A]` ⭐

**Зачем:** агент = LLM, которая может вызывать твой код/API и действовать. Горячая тема и сильный пет-проект.

**4.6.1 Function calling / tool use** ⭐ — схема инструмента, цикл «модель просит инструмент → код выполняет → результат назад». 🎯 *Докажи себе:* дай модели инструмент «калькулятор» и заставь её им воспользоваться.

**4.6.2 ReAct-паттерн** — чередование рассуждения и действий. 🎯 *Докажи себе:* объясни шаги ReAct на примере «найди погоду и реши, брать ли зонт».

**4.6.3 Агентные циклы, планирование, память** — многошаговые задачи, краткосрочная/долгосрочная память. 🎯 *Докажи себе:* собери агента, делающего 2–3 инструментальных вызова для одной цели.

**4.6.4 Мульти-агентные системы (обзор)** — оркестратор + специализированные агенты. 🎯 *Докажи себе:* опиши, какие роли ты бы выделил в агенте-ассистенте по почте.

**4.6.5 Надёжность** — лимит шагов, обработка ошибок инструментов, человек-в-цикле. 🎯 *Докажи себе:* добавь защиту «не больше N шагов» и обработку упавшего инструмента.

**4.6.6 Безопасность инструментов** ⭐ — не давать необратимых действий без подтверждения; данные из инструментов — это данные, а не команды. 🎯 *Докажи себе:* сделай так, чтобы «опасное» действие требовало явного подтверждения пользователя.

**Ресурсы:** Prompt Engineering Guide — раздел про агентов (EN); DeepLearning.AI «Functions, Tools and Agents» / «AI Agents in LangGraph» (EN); docs провайдеров по tool use (EN, первоисточники); Claude/OpenAI Cookbooks (EN).

**Контрольное по блоку 4.6:** собери агента с 2–3 инструментами (например, поиск + калькулятор + вызов твоего API), с лимитом шагов и подтверждением необратимых действий.

## Блок 4.7 — MCP (Model Context Protocol) `[A]`

**Зачем:** открытый стандарт подключения моделей к инструментам и данным («USB-C для ИИ»), быстро становится индустриальной нормой. Твой опыт API/интеграций здесь прямо в кассу.

**4.7.1 Проблема N×M и идея MCP** — почему отдельные коннекторы не масштабируются. 🎯 *Докажи себе:* объясни, какую проблему интеграций решает единый протокол.

**4.7.2 Архитектура** — host / client / server, JSON-RPC, сессии. 🎯 *Докажи себе:* нарисуй, кто кому шлёт сообщения в MCP.

**4.7.3 Примитивы** — tools (действия), resources (данные только для чтения), prompts (шаблоны). 🎯 *Докажи себе:* приведи по примеру каждого примитива.

**4.7.4 Готовые серверы** — подключение существующих (файлы, Postgres, GitHub) к MCP-хосту. 🎯 *Докажи себе:* подключи готовый MCP-сервер к клиенту и вызови инструмент.

**4.7.5 Свой MCP-сервер** ⭐ — экспонирование своего API как MCP-сервера (твой бэкенд-навык). 🎯 *Докажи себе:* напиши минимальный MCP-сервер с одним инструментом и протестируй его.

**Ресурсы:** официальная документация MCP — *первоисточник, EN, бесплатно.* https://modelcontextprotocol.io/docs/getting-started/intro · спецификация и SDK: https://github.com/modelcontextprotocol — *проверено на июнь 2026.*

**Контрольное:** напиши свой MCP-сервер, оборачивающий одну полезную функцию, и подключи его к MCP-хосту.

## Блок 4.8 — Оценка качества (eval) `[A]` ⭐

**Зачем:** без eval разработка LLM-приложений слепа. Здесь оживают метрики (Этап 2.2) и статистика (Этап 1.4). Eval-навык сильно отличает инженера от «крутильщика промптов».

**4.8.1 Зачем eval** — почему «на глаз» не работает, регрессии при смене модели/промпта. 🎯 *Докажи себе:* поймай случай, где «улучшение» промпта ухудшило результат на части примеров.

**4.8.2 Offline eval** — тестовые наборы, эталонные ответы, золотой датасет. 🎯 *Докажи себе:* собери набор из 20 пар «вопрос → ожидание» для своего RAG.

**4.8.3 Метрики RAG** — контекстная релевантность, faithfulness (верность контексту), answer relevance. 🎯 *Докажи себе:* оцени свой RAG по faithfulness и найди галлюцинацию.

**4.8.4 LLM-as-a-judge** ⭐ — модель как оценщик, его плюсы и подвохи (смещения). 🎯 *Докажи себе:* напиши промпт-судью, оценивающий ответ по шкале с обоснованием.

**4.8.5 Инструменты** — RAGAS, TruLens, LangSmith, простые самописные харнессы. 🎯 *Докажи себе:* прогони RAGAS на своём RAG и получи числа.

**4.8.6 Онлайн-метрики и A/B** — логирование, обратная связь пользователей, A/B (Этап 1.4.6). 🎯 *Докажи себе:* опиши, как ты сравнил бы две версии промпта статистически корректно.

**Ресурсы:** Prompt Engineering Guide — раздел про eval и LLM-as-a-judge (EN); RAGAS docs (EN); DeepLearning.AI «Evaluating AI Agents» (EN). — *проверено на июнь 2026.*

**Контрольное по блоку 4.8:** построй eval-харнесс для своего RAG (золотой набор + RAGAS + один LLM-судья) и зафиксируй базовые числа; покажи, что улучшение в блоке 4.5 реально подняло метрику.

## Блок 4.9 — Фреймворки и оркестрация `[A]`

**Зачем:** ускоряют сборку, но добавляют абстракций. Важно понимать, когда они помогают, а когда «голый» SDK чище.

**4.9.1 LangChain / LangGraph** — что это, граф-агенты (LangGraph), типичная критика «слишком много магии». 🎯 *Докажи себе:* собери один и тот же мини-пайплайн на LangChain и на чистом SDK, сравни читаемость.

**4.9.2 LlamaIndex** — заточен под RAG/работу с данными. 🎯 *Докажи себе:* собери RAG на LlamaIndex и сравни с ручным из блока 4.5.

**4.9.3 Vercel AI SDK** ⭐ — *TypeScript-first*, стриминг, generative UI — прямо под твой фронтенд-стек. 🎯 *Докажи себе:* выведи стриминговый ответ модели в React-компонент через AI SDK.

**4.9.4 Альтернативы** — Semantic Kernel, Haystack, чистый SDK провайдера. 🎯 *Докажи себе:* сформулируй правило «когда брать фреймворк, а когда нет».

**4.9.5 Актуальность** — экосистема меняется каждые месяцы; не вмораживайся в один фреймворк, держи логику отделённой. 🎯 *Докажи себе:* объясни, как изолировать бизнес-логику от конкретного фреймворка.

**Ресурсы:** официальные docs LangChain/LangGraph, LlamaIndex, **Vercel AI SDK** (`ai-sdk.dev` / `sdk.vercel.ai`) — EN, первоисточники; DeepLearning.AI «LangChain for LLM Application Development» (EN). — *проверено на июнь 2026.*

## Блок 4.10 — Дообучение (опционально) `[B]`

**Зачем:** когда RAG и промптов не хватает (узкий домен, стиль, малоресурсный язык). Для старта карьеры **не обязательно** — RAG+агенты покрывают большинство кейсов.

**4.10.1 Когда fine-tuning** — честный чек-лист «сначала попробуй промпт/RAG». 🎯 *Докажи себе:* приведи кейс, где fine-tuning оправдан, и кейс, где это лишнее.

**4.10.2 LoRA / QLoRA / PEFT** — дешёвое дообучение малой части весов. 🎯 *Докажи себе:* объясни, почему LoRA дешевле полного дообучения.

**4.10.3 Instruction tuning (обзор)** — превращение базовой модели в «послушную». 🎯 *Докажи себе:* объясни разницу датасетов для pretraining и instruction tuning.

**4.10.4 Где обучать** — ⚠️ нужна GPU; учебная LoRA на малой модели влезает в бесплатный Colab, серьёзное — платная аренда. ☁️ 🎯 *Докажи себе:* дообучи LoRA на маленьком датасете в Colab и сравни с базовой моделью.

**Ресурсы:** Hugging Face TRL/PEFT docs (EN); DeepLearning.AI курсы по fine-tuning (EN); Unsloth (быстрый LoRA) — EN.

**Контрольное (опц.):** дообучи LoRA на узкую задачу в Colab и покажи прирост над промпт-решением через eval из блока 4.8.

## 🏁 Проект-веха Этапа 4 — твой флагманский пет-проект

**Full-stack AI-приложение с настоящим интерфейсом.** Это центральный экспонат портфолио, где фронтенд — твоё преимущество.

Минимальный состав:
1. **Frontend на React** (твой конёк): чат со стримингом, загрузка документов, отображение источников/цитат, приятный UX.
2. **Backend с RAG**: индексация документов → векторная БД (pgvector/Qdrant) → извлечение с re-ranking → генерация с цитированием.
3. **Хотя бы один агентный инструмент** (например, поиск в интернете или вызов внешнего API) с tool use.
4. **Eval-харнесс**: золотой набор + RAGAS, чтобы доказать качество цифрами.
5. (бонус) **MCP-сервер**, оборачивающий часть функциональности.

*Критерий «сделано»:* приложение работает end-to-end, отвечает по твоим данным с цитатами, есть измеримое качество, и оно задеплоено (деплой — в Этапе 5). Это ровно то, что показывают на собеседовании на AI Engineer.

**Лёгкий [A]-минимум:** работающий RAG-чат с цитатами и базовым eval — уже сильный пункт; агента и MCP можно добавить во вторую итерацию.
