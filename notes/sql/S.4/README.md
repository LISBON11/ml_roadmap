# S.4 — Соединения (JOIN)

> Пример заполненной заметки — как ориентир формата. Остальные создавай по `notes/_шаблон.md`.

## Суть простыми словами
JOIN склеивает строки из двух таблиц по условию (обычно по ключу). Разница между видами — что делать со строками, у которых нет пары.

## Ключевое
- **INNER JOIN** — только совпавшие пары (пересечение).
- **LEFT JOIN** — все строки левой таблицы + совпавшие из правой; где пары нет → `NULL`. Классика для «найди тех, у кого НЕТ связанных записей» (`LEFT JOIN ... WHERE right.id IS NULL`).
- **RIGHT JOIN** — зеркально LEFT (на практике почти всегда переписывают как LEFT).
- **FULL JOIN** — все строки обеих таблиц, несовпавшие с `NULL`.
- Условие — в `ON`; фильтр правой таблицы в `WHERE` **превращает LEFT в INNER** (частая ловушка — фильтр по правой таблице клади в `ON`).

## Пример
```sql
-- средняя латентность по каждой модели, включая модели без прогонов
SELECT m.name, AVG(r.latency_ms) AS avg_latency
FROM models m
LEFT JOIN runs r ON r.model_id = m.model_id
GROUP BY m.name;
```

## Грабли
- Фильтр `WHERE r.status = 'ok'` при LEFT JOIN незаметно выкидывает модели без прогонов → используй `AND r.status='ok'` в `ON`.
- Забыть `GROUP BY` при агрегате после JOIN.

## Псевдонимы и неоднозначность (из Практикума)
- Поле, которое есть в **обеих** таблицах, без указания таблицы → `ERROR: column reference "city_id" is ambiguous`. Пиши `users.city_id` / псевдоним.
- Псевдонимы таблиц: `FROM users AS u JOIN cities AS c ON u.city_id = c.city_id`.
- `USING (city_id)` — короткая замена `ON u.city_id = c.city_id`, когда поле называется одинаково (и оставляет один столбец в выводе).
- `RIGHT JOIN` = зеркало LEFT; полезен, когда основная по смыслу — правая таблица (не переписывать старый запрос). `LEFT/RIGHT/FULL OUTER JOIN` — синонимы `LEFT/RIGHT/FULL JOIN`.
