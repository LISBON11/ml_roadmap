# _разное — вводные заметки без привязки к коду трекера

> Сюда собраны заметки Практикума, которые не ложатся на конкретный код карты: обзорное «Введение в ИИ», основы устройства реляционных БД и вводные про ML. Коротко, как справка. Мусорные обрывки заметок Практикума (пустые/оборванные фразы) выброшены.

## Введение в ИИ

### Как работают нейросети / LLM
- Сеть = формула из **слоёв**, выполняемых по очереди (как конструктор). `Linear` (полносвязный) ловит простые зависимости; `ReLU` (нелинейный) — сложные, отсекая по порогу. Порядок слоёв = **архитектура**.
- Под тип данных — своя архитектура (нет универсальной): картинки — свои слои, текст — свои. Известные: Midjourney (картинки), GPT (текст).
- **Обучение** (цикл): инициализировать веса (0/1/случайные) → подставить примеры с известными ответами → посчитать **ошибку** (разница с правильным) → сдвинуть веса, чтобы ошибку уменьшить → повторять, пока ошибка не станет малой.
- **GPT** = Generative Pre-trained Transformer. Генерирует текст **по одному токену** (токен ≈ слово), предсказывая **вероятности** следующего слова. Токены → числовые векторы (**эмбеддинги**), хранящие смысл. **Self-attention**: трансформер смотрит сразу на все прошлые слова с разной «важностью».
- Три этапа обучения LLM: **предобучение** (учит язык, как продвинутый Т9, команды не понимает) → **SFT / обучение на инструкциях** (пары «запрос → эталонный ответ», файнтюнинг) → **обучение с подкреплением** (без эталона, но с оценкой хорошо/плохо — учится отвечать «по-человечески», без грубости/неполноты).

### Безопасность и этика
- ИИ не понимает контекст, эмоции, подтекст — выдаёт шаблон по словам; совет может навредить (пример: «подари подарок» при конфликте из-за денег).
- Авторское право (практика США): если произведение создано в основном нейросетью без творческого вклада человека — охраны нет (дело *Zarya of the Dawn*: защитили только написанное человеком). Правообладатели судятся с ИИ-сервисами за обучение на их материалах без разрешения.

## Основы реляционных БД (PostgreSQL)

### Витрина данных (Data Mart)
- Заранее подготовленная таблица со всеми нужными данными под задачу/модель (без длинных JOIN каждый раз).
- Создание: `CREATE TABLE имя AS SELECT ...` — результат `SELECT` **физически** сохраняется в новую таблицу. Удалить: `DROP TABLE имя`.
- **Витрина не обновляется сама** — это «снимок» на момент создания. Обновляют: пересозданием, по расписанию (Airflow/dbt) или через `MATERIALIZED VIEW`.
- Нужна, когда: расчёты повторяются, данных много, работает команда, важна воспроизводимость. Избыточна для разовых задач и данных «в реальном времени».

### Изучение незнакомой БД (`information_schema`)
- Схемы: `SELECT schema_name FROM information_schema.schemata;`
- Таблицы: `... information_schema.tables WHERE table_schema = '<схема>';`
- Поля и типы: `... information_schema.columns ...`; ключи: `... key_column_usage ...` (`_pkey`/`_fkey` в `constraint_name`).
- Системные таблицы PostgreSQL — с префиксом `pg_` в схеме `pg_catalog`. По умолчанию таблицы кладутся в схему `public` (её имя можно не указывать).

### Схемы, ключи, связи, нормализация
- **Схема** — контейнер связанных таблиц; помогает структурировать, разграничивать доступ и избегать конфликта имён (`online_store.orders` vs `physical_store.orders`). Указывай схему перед таблицей.
- Первичный ключ: **естественный** (номер паспорта), **суррогатный** (сгенерированный id, не меняется), **составной** (несколько столбцов вместе).
- **Многие-ко-многим** реализуют через промежуточную (join-)таблицу с двумя внешними ключами.
- **Нормализация** — организация данных для минимума избыточности и аномалий обновления. Связь «один-к-одному» иногда разбивают на две таблицы (читаемость, производительность, доступ).
